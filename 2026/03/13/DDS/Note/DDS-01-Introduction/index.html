DDS-01: Introduction

DDS-01: Introduction

这份课件主要涵盖了三个核心方面：课程范围、Dependability (可信性) 的核心动机，以及通过 Apollo 13 案例说明的故障模型假设的重要性。

1. Course Scope (课程范围)

本课程分为三条核心主线：

  • Dependable Systems (可信系统):
    • 讨论如何定义可信系统，以及如何对其进行建模 (Modeling)、分析 (Analysis) 和评估 (Evaluation)。
  • Distributed Algorithms (分布式算法):
    • 讨论分布式环境下的经典协调问题，如 Leader Election (领导者选举)、Consensus (共识) 和 Byzantine Agreement (拜占庭协定)。
  • Synchronisation (同步):
    • 讨论分布式系统中的时间概念，包括 Physical Clock Synchronisation (物理时钟同步)、Logical Clocks (逻辑时钟) 和 Vector Clocks (向量时钟)。

这门课不仅关注“系统为何失效”，更关注在设计阶段如何避免失效，以及在失效发生后如何维持系统协调。

2. Definition of Dependability (可信性定义)

课程给出的官方定义如下：

Dependability: A system is said to be dependable if it is trustworthy enough for reliance to be justifiably placed on the services it delivers.

定义的关键点在于 Justifiably (可论证地)：

  • Dependability 不是主观的感觉，而是一种基于证据的信任。
  • 这种“可论证性”通常依赖于明确的 Specification (规格说明)、System Model (系统模型) 和 Fault Model (故障模型)。
  • 它是一种可以被量化、分析和评估的系统属性。

3. Motivation for Dependability (动机)

3.1 Beyond Safety-Critical Systems

虽然 Safety-critical systems (如航空电子 Avionics、核反应堆 Nuclear Reactors、线控系统 X-by-wire) 对可信性有极致要求，但老师强调，现代消费级系统也越来越重视 Dependability，因为用户对 QoS (Quality of Service) 的容忍度正在降低。

3.2 Murphy’s Law as a Design Principle

Murphy’s Law: “If something can go wrong, it will go wrong.”

在系统设计中，这句话被视为一种工程方法论：

  • Design for Failure: 不要假设环境是完美的。
  • Core Questions:
    1. 什么会坏？ (What can fail?)
    2. 坏了之后系统能否继续提供正确且及时的服务？ (Can the system remain correct and timely?)

4. Redundancy: The Basic Means (冗余：基本手段)

实现 Dependability 的核心手段是 Redundancy (冗余)，通常分为两种形式：

4.1 Space Redundancy (空间冗余)

  • Concept: 同一时间准备多个资源。
  • Examples: 多副本存储 (Replicated storage)、多台机器、硬件冗余模块。

4.2 Time Redundancy (时间冗余)

  • Concept: 同一个任务换个时间再执行一次。
  • Examples: 重试机制 (Retrying)、回退后重新运行 (Rollback and restart)、检查点机制 (Checkpointing)。

这些技术在后续的 Software Fault Tolerance 章节中会详细讨论。

5. Case Study: Apollo 13 (案例分析)

Apollo 13 案例在课程中用于揭示系统事故的深层原因：Incorrect Failure Assumptions (错误的故障假设)。

5.1 Root Causes (根源)

  • Pre-launch issues: 液氧罐泄漏被忽略，由于进度压力 (Schedule pressure) 导致工程师未能深入调查。
  • Design Flaws: 某些导线在测试中受损，绝缘层接近失效。
  • Fault Propagation (故障传播): 电弧引发起火，导致氧气压力异常，进而损毁燃料电池，切断了电力、水和氧气的供应。

5.2 Key Lessons (核心教训)

  • Failure Chain: 故障不是孤立的，它会跨组件传播，形成 Fault -> Error -> Failure 链条。
  • Independence Assumption: 设计者错误地假设了故障的独立性 (Assumed independence of failures)。
  • Probability Misjudgment: 错误地认为多重故障同时发生的概率极低 (Assumed low probability of multiple failures)。

6. Course Framework (课程后续框架)

所有的分析都建立在以下三要素之上：

  • System Model: 我们如何观察系统。
  • Fault Model: 我们假设什么会出错。
  • Specification: 什么是“正确”的行为。

这些概念将在下一章 Dependability Concepts 中正式定义。

7. Summary for Revision (复习要点)

  • Dependability Definition: Focus on “justifiably placed reliance”.
  • Three Pillars: Dependable systems, Distributed algorithms, Synchronisation.
  • Redundancy Types: Space and Time.
  • Apollo 13 Insight: Danger of incorrect failure assumptions, importance of fault propagation and isolation.
