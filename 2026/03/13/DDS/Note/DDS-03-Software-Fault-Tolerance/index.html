DDS-03: Software Fault Tolerance

DDS-03: Software Fault Tolerance

这一讲的核心问题是：给定一个满足规格的程序 P，在故障模型 F 下，怎样把它变成仍然可接受的程序 P'？

1. Fault Tolerance Classes (容错等级)

课件将软件容错建立在 Specification 的两个基本维度上：

  • Safety: 坏事永远不要发生 (Bad things never happen)。
  • Liveness: 好事最终会发生 (Good things eventually happen)。

根据对这两个属性的满足程度，容错分为以下几类：

1.1 Masking Fault Tolerance (屏蔽容错)

  • Definition: 程序在故障 F 下仍满足完整的 SPEC (同时满足 Safety 和 Liveness)。
  • Key Idea: 故障被“屏蔽”了。从外部看，系统运行完全正常，服务未中断。这是最强的一类容错。

1.2 Fail-safe Fault Tolerance (失效安全)

  • Definition: 程序在故障出现后只保证 Safety，不保证 Liveness。
  • Key Idea: 宁可停下来 (Terminate/Halt)，也不要做错。重点是“保命”。
  • Application: 工业保护、刹车系统。

1.3 Non-masking Fault Tolerance (非屏蔽容错)

  • Definition: 程序在故障下保证 Liveness，但在恢复过程中可能暂时违反 Safety。
  • Key Idea: 允许先进入错误状态，但最终必须能恢复到正确轨道。

1.4 F-intolerant (不容错)

  • Definition: Safety 和 Liveness 都不能保证。

2. Fundamental Components: Detectors & Correctors (检测器与修正器)

要满足上述容错等级，系统需要两类基础构件：

2.1 Detector (检测器) —— 守护 Safety

  • Definition: “A detector is a class of program components that asserts the validity of a predicate in a running program.”
  • Logic: if (NOT PREDICATE) then ERROR;
  • Impact: 能够识别风险并触发警报，但通常会导致程序终止，从而牺牲 Liveness。
  • Implementation (Run-time Checks):
    • Replication checks: 比较副本输出。
    • Timing checks: 计时器/看门狗 (Watchdog)。
    • Coding checks: 奇偶校验、汉明码。
    • Reasonableness checks: 逻辑合理性检查（如：年龄不能为负）。

2.2 Corrector (修正器) —— 挽救 Liveness

  • Definition: “A corrector is a class of program components that imposes a given predicate on a running program.”
  • Logic: if (NOT SAFETY) then enforce predicate;
  • Impact: 在错误发生后干预，强制将系统拉回正常状态。确保了 Liveness，但允许暂时违反 Safety。
  • Implementation: Exception Handlers (Interface, Local, Failure levels)。

2.3 Necessary and Sufficient Conditions (充要条件)

  • Detectors 是设计 Fail-safe 的充要条件。
  • Correctors 是设计 Non-masking 的充要条件。
  • Detectors + Correctors 是设计 Masking 的充要条件。

3. Error Recovery: Forward vs. Backward (前向与后向恢复)

3.1 Backward Error Recovery (后向恢复)

  • Mechanism: 当检测到错误时，回滚 (Rollback) 到之前记录好的安全状态 (Checkpoint)，然后重启。
  • Key Tools: Checkpointing。

3.2 Forward Error Recovery (前向恢复)

  • Mechanism: 检测到错误后，不回头，直接通过补救措施将系统推到一个全新的无错状态。
  • Key Tools: Recovery Blocks (某些方面体现前向特征)。

4. Checkpointing & Recovery Line (检查点与恢复线)

在多进程分布式环境中，回滚必须回到一个一致的历史切面。

4.1 Recovery Line (恢复线)

  • Definition: “A set of checkpoints across all processes to which the programs can be rolled back to ensure a consistent error-free state.”

4.2 Consistency Rules (一致性准则)

一条恢复线是否一致，取决于它对消息的处理：

  1. OK: 消息已发送但尚未接收。
  2. NOT OK: 消息已接收但历史上尚未发送（产生“孤儿消息”）。
  • Rule: 沿着恢复线，不能出现只有接收 (Receive) 而没有对应发送 (Send) 的情况。

4.3 Domino Effect (多米诺效应)

如果检查点打得不一致，回滚会导致连锁反应，最终可能迫使所有进程退回程序开始的最早时刻。

5. Recovery Blocks (RB, 恢复块)

恢复块是一种利用软件版本冗余来实现容错的技术，结合了空间和时间冗余。

5.1 Structure (结构)

  1. Capture Checkpoint: 记录初始状态。
  2. Primary Module: 执行主版本。
  3. Acceptance Test (AT): 验证结果。
  4. Restore & Retry: 如果 AT 失败，恢复 Checkpoint 并执行 Secondary Module。

5.2 The Critical Role of Acceptance Test (验收测试)

  • Challenge: 设计一个既准确又不过于复杂的 AT 非常困难。
  • Risk: 如果 AT 不可靠（False Positive/Negative），整个系统的可靠性甚至可能低于只用一个高质量主版本的系统。

6. N-Version Programming (NVP, N-版本程序设计)

NVP 是一种通过多版本并行执行和结果比对来实现容错的技术。

6.1 Core Mechanism

  • Definition: “Independent generation and utilisation of N functionally equivalent programs, where N >= 2.”
  • Parallel Execution: 多个独立开发的版本同时运行，通过 Majority Voting (多数表决, $N \ge 3$) 决定输出。
  • Detection ($N=2$): 两个版本结果不一致时触发恢复或报警。

6.2 NVP vs. Recovery Blocks

Feature Recovery Blocks (RB) N-Version Programming (NVP)
Execution Sequential (顺序) Parallel (并行)
Redundancy Space + Time Space
Hardware 1 Channel N Channels
Decision Acceptance Test Majority Voting

6.3 The NVP Axiom & Reality (公理与现实)

  • The Axiom: NVP 假设各个版本的失效是完全独立的 (Independent failures)。
  • The Reality: UVA/UCI 实验表明，版本之间往往存在相关性错误 (Correlated errors)。不同程序员在面对同一复杂 Specification 时，容易在相同的地方犯错。因此，NVP 的实际效果往往低于理论预期。

7. Fault Injection & Coverage (故障注入与覆盖率)

  • Fault Injection Analysis: 主动施加异常条件，观察系统行为并评估容错机制。
  • Coverage (覆盖率):
    $$Coverage = \frac{\text{Successfully handled cases}}{\text{All subjected test cases}}$$
    • $Coverage = 1$: 完全容错。
    • $Coverage = 0$: 完全无防御。

8. Summary for Revision (复习要点)

  • FT Classes: Masking (Both), Fail-safe (Safety), Non-masking (Liveness).
  • Detectors/Correctors: Guarding Safety vs. Saving Liveness.
  • Consistency Rule: No receive without send on the recovery line.
  • RB vs. NVP: Sequential + AT vs. Parallel + Voting.
  • NVP Limitation: Independence assumption often fails in practice (Correlated failures).
