DDS-02: Dependability Concepts

DDS-02: Dependability Concepts

Definition: A system is said to be dependable if it is trustworthy enough for reliance to be justifiably placed on the services it delivers.
Dependability means the system is worthy of your trust when it provides services.

1. Fundamental Models (基础模型)

为了分析系统的可信性，我们必须建立以下模型：

1.1 System Model (系统模型)

系统模型定义了我们在哪个抽象层级观察系统：

  • Elements: 系统由硬件 (Hardware)、软件 (Software) 甚至人 (Human) 组成。
  • Interaction: 通过定义的接口 (Interface) 进行通信。
  • Recursive Nature: 子系统本身也可以被视为一个 Element（递归性）。
  • Service: 用户感知的行为。必须满足 Correct (正确) 和 Timely (及时)。

!02-1.png

1.2 Fault Model (故障模型)

定义了潜在的问题：我们假设什么东西会出错？（如：Bit flip 比特翻转、Process crash 进程崩溃）。

1.3 Specification (规格说明)

系统行为的“神谕” (Oracle)。没有 Specification，就无法定义什么是 Failure (失效)。

2. The Dependability Tree (分类树)

可信性由三个分支组成：

  1. Impairments (损伤): 敌人（Fault, Error, Failure）。
  2. Means (手段): 武器（Prevention, Removal, Forecasting, Fault Tolerance）。
  3. Attributes (属性): 衡量标准（Reliability, Availability, Safety, etc.）。

!02-2.png

3. Impairments: The Fundamental Chain (损伤与因果链)

系统思维的核心在于：底层的“失效 (Failure)”就是上层的“故障 (Fault)”。

3.1 The Recursive Nature of Causality (因果递归)

整个系统类似于俄罗斯套娃，每一层都有自己的 Fault -> Error -> Failure 循环：

  • Layer N (Lower Level, e.g., Hardware):
    • Fault: 宇宙射线击中内存，导致一个比特翻转 (Bit flip)。
    • Error: 内存中的数据状态变成了错误的数值（内部状态损坏）。
    • Failure: 硬件单元输出了错误的计算结果（输出边界失效）。
  • Layer N+1 (Higher Level, e.g., Software):
    • Fault: 接棒时刻！ 底层的 Failure 成为该层的 Fault（错误输入或坏环境）。
    • Error: 函数内部变量被赋值为错误的值（坏内部状态）。
    • Failure: 函数返回了错误的值给主程序（该层失效）。

3.2 Relativity (相对性)

“故障”还是“失效”，取决于你站在哪一层 (Abstraction Level) 看问题：

  • 芯片设计师眼中：这是我的 ALU Failure (失效) 了。
  • 软件工程师眼中：这是我的程序遇到了一个 Hardware Fault (硬件故障)。

3.3 Summary of the Chain

Failure at one level $\to$ Fault at a higher level $\to$ Error in internal state $\to$ (if not tolerated) $\to$ Failure at the higher level.

!02-3.png

4. Dependability Means (可靠性手段)

如果说“损伤”是敌人，那么“手段”就是我们手中的武器。

4.1 Fault Prevention (故障预防)

旨在阻止故障的发生、引入和传播。

Intention: To hinder and obstruct the occurrence, introduction and spread of faults.

  • Techniques: 模块化设计 (Modular design)、Scrum、敏捷开发 (Agile)、瀑布流模型 (Waterfall) 等流程控制。

4.2 Fault Removal (故障移除)

旨在系统中减少已存在的故障数量。

  1. Validation (验证): Whether a system adheres to a set of defined properties. (系统是否符合定义属性？)
  2. Diagnosis (诊断): Identify faults preventing these properties from being fulfilled. (识别导致问题的具体故障。)
  3. Correction (修正): Modify the system to eliminate the fault. (修改系统消除故障。)
  • Techniques: Debugging、代码审查 (Code Reviews)、软件测试 (Software Testing)。

4.3 Fault Forecasting (故障预测)

一种评估手段 (Assessment strategy)，旨在估计系统中故障的数量和后果。

Estimate potential failures and consequences in the system.

  • Techniques: 故障注入分析 (Fault Injection Analysis)。通过模拟故障来评估系统的覆盖率 (Coverage)。

4.4 Fault Tolerance (故障容错)

First among equals (同级中的首要)。承认系统不可能 100% 无故障。

  • Mechanism:
    1. Detection: 识别错误状态（Detectors）。
    2. Recovery: 恢复状态（Correctors），恢复到正确或安全状态。
  • Associated Techniques: N-Version Programming (NVP)、Recovery Blocks、Checkpointing。

5. Dependability Attributes (可靠性属性)

属性是衡量一个系统是否“值得信赖”的具体度量标准。

5.1 Reliability (可靠性) - $R(t)$

定义：系统在指定的时间周期内提供正确服务的概率。

  • Mathematical Definition:
    $$R(t) = P(\text{Service correct in } [t_0, t] \mid \text{Service correct at } t_0)$$
  • Unreliability (不可靠性): $Q(t) = 1 - R(t)$。

5.2 Availability (可用性) - $A(t)$

定义：系统在给定时刻能够正确提供指定服务的概率。

  • Steady-state Availability (稳态可用性):
    $$A_{steady} = \frac{MTTF}{MTTF + MTTR}$$
  • MTTF (Mean Time To Failure): 平均失效前时间。 ($1/\lambda$)
  • MTTR (Mean Time To Repair): 平均修复时间。 ($1/\mu$)

“The Nines” (商业指标):

Availability (%) Downtime per Year (approx.)
99.9% (3 Nines) 8.76 hours
99.999% (5 Nines) 5.26 minutes

5.3 Maintainability (可维护性) - $M(t)$

定义：失效的系统在时间 $t$ 或更短时间内被修复的概率。

  • Function: $M(t) = 1 - e^{-\mu t}$ (假设修复率 $\mu$ 恒定)。

5.4 Safety (安全性)

安全性反映了系统在不损坏或危及周围环境的情况下运行的程度。

  • Safe States: 系统通常具备“安全状态”（如铁路信号灯全红）。虽然服务中断，但环境安全。

5.5 Security (保障/安保)

在可靠性分类法中，Security 是一个综合概念，由 CIA 三元组 构成：

  1. Confidentiality (机密性): 防止信息向未经授权的实体泄露。
  2. Integrity (完整性): 确保系统不存在不当的变更（防止未经授权的修改）。
  3. Availability (可用性): 确保授权用户可以访问服务。

6. Summary for Revision (复习重点)

  • Recursive Chain: Fault -> Error -> Failure -> Fault (next level).
  • Means Hierarchy: Prevention (stop before), Removal (clean existing), Forecasting (estimate), Tolerance (live with).
  • MTTF/MTTR/MTBF: Know how to calculate Availability.
  • Reliability vs. Availability: Continuity vs. Instantaneous state.
