Requirements Engineering for AI-Enabled Systems (AI赋能系统的需求工程)

需求工程 (Requirements Engineering, RE) 是软件生命周期中不可或缺的阶段，其核心在于理解系统目标、用户需求以及系统应交付的功能。对于AI系统而言，需求工程更为复杂，因为除了传统功能外，还必须充分考虑机器学习模型 (ML models)和数据 (Data)。

为什么AI需要特殊的需求工程？

许多AI系统的失败并非因为代码bug，而是由于对环境假设 (Assumptions) 和真实世界需求 (Requirements) 的误解与遗漏。

经典失败案例及启示：

  1. Amazon自动化招聘 (Automated Hiring)：使用历史数据训练导致强烈的性别偏见。
    • 启示：在写需求时，必须将数据视为“一等公民”，这表明训练数据的质量、偏差检测和代表性是AI需求工程的重要考量。
  2. Tesla自动驾驶 (Autopilot in Vehicles)：驾驶员在方向盘上放重物欺骗安全系统。
    • 启示：系统未能准确区分“真实人手”与“无生命物体”。这属于未充分考虑安全护栏 (Safeguards) 和环境互动动态的假设遗漏。
  3. IBM Watson医疗推荐 (IBM Watson)：系统推荐了当地不可用的治疗方案，或不符合当地医疗方案的协议。
    • 启示：AI系统的需求没有考虑本地化假设 (Locality assumptions) 和医疗合规性。系统必须在其所操作的具体环境背景下进行假设和验证。

核心知识点模块

为了更好地理解AI系统的需求工程，我们需要深入了解以下几个子模块：

  1. subnote_Untangling_Goals_in_AI：理解并梳理组织、系统、用户和模型层面的目标，并进行细化。
  2. subnote_World_vs_Machine_REQ_ASM_SPEC：区分真实世界与软件机器，理解需求 (REQ)、规范 (SPEC) 与假设 (ASM) 之间的逻辑关系。
  3. subnote_Non_Functional_Requirements_for_AI：AI系统特有的非功能性需求（如公平性、解释性及数据质量控制）。improtant
  4. subnote_Requirements_Elicitation_and_Negotiation：如何有效获取并协商AI系统的需求，防范过度工程。

系统级思考 (Systems Thinking)

软件不是在真空中运行的，机器学习仅仅是整个系统中的一个组件。系统级思考对需求工程师至关重要：

  • 整体方法 (Holistic approach)：考虑更大的图景，涉及所有潜在的利益相关者。
  • 反馈循环 (Feedback loops)：智能系统设计中存在持续的反馈。这些反馈可能来自系统操作的用户、系统本身或环境（如自动驾驶中环境的实时变化）。数据生成的反馈如果不加以处理和理解，极易导致系统偏见。
  • 必须理解人类如何与系统交互，并处理这些互动。

“The hardest single part of building a software system is deciding precisely what to build… No other part of the work so cripples the resulting system if done wrong.” —— Fred Brooks, Mythical Man Month (1975)

构建软件系统最困难的部分就是准确地决定要构建什么。如果在设计阶段构建了错误的系统，后期修复的成本将极其高昂（Think before coding）。
