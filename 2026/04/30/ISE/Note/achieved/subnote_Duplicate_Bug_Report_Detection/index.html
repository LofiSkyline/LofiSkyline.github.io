Duplicate Bug Report Detection (重复缺陷报告检测)

Duplicate Bug Report Detection (重复缺陷报告检测)

  • 目的 (Purpose): 检测给定的报告是否以前已经被处理过 (has already been dealt with before)。
  • 原因 (Why?): 节省工作量！没有必要浪费时间去调查已经处理过的事情。

问题表述的转变

为什么不直接使用二分类模型预测一个报告是否是重复的？
因为直接进行二分类的准确率太低 (accuracy is too bad)。
因此，问题的表述被放宽：给定一个新的缺陷报告，系统推荐一个包含潜在相似历史报告的列表 (recommend a list) 供开发人员人工检查。

基于 BM25Fext 的经典方法

Sun 等人提出了基于 BM25F 相似度度量的经典工作。

数据结构：桶 (Buckets)

  • 缺陷库被组织成一个列表的桶 (Buckets)（类似哈希表结构）。
  • 每个桶的键是一个主报告 (master report)，其值是同一个缺陷的所有重复报告列表。
  • 每个桶代表一个独特的缺陷 (distinct bug)。

工作流程

  1. 当提交一个新鲜的缺陷报告 Q 时，系统计算 Q 与每个桶之间的相似度 (similarity)。
  2. 返回具有前 K 个相似度的桶对应的主报告。
  3. 报告 Q 与某个桶的相似度，被定义为报告 Q 与该桶中每一个报告的相似度中的最大值 (maximum similarity)。

相似度计算与特征

  • BM25Fext: 扩展了传统的 BM25F 指标以处理较长的查询。
  • 综合了7个维度的相似度特征：摘要 (summary)、描述 (description)、产品 (product)、组件 (component)、类型 (type)、优先级 (priorities) 和版本 (versions)。
  • 参数优化: 该度量公式中包含大量权重参数。作者构建了一个包含查询报告、相关重复报告和不相关报告的数据集，并使用梯度下降 (gradient descent) 算法来优化 BM25Fext 的参数，以达到最佳推荐效果。
