Intelligent Data-Driven Bug Report Analysis (智能数据驱动的缺陷报告分析)

智能软件工程 (Intelligent Software Engineering, ISE) 旨在将人工智能(AI)技术应用于软件工程任务，以提高软件系统的开发和维护效率。本节课主要聚焦于如何利用AI进行缺陷报告分析 (Bug Report Analysis)。

什么是缺陷报告？(What is a Bug Report?)

缺陷报告是由开发人员、测试人员和最终用户收集并记录软件缺陷的软件制品 (Software artifact)。随着开源软件的快速发展，产生了海量的缺陷报告。

  • 常见的缺陷跟踪系统 (Issue/report trackers)：GitHub, JIRA, HubSpot。
  • 缺陷报告通常包含：标题 (Title)、描述 (Description)、代码片段 (Code snippets)、评论 (Comments) 和标签 (Labels)。

为什么要分析缺陷报告？(Why Analyze Bug Reports?)

  1. 数量庞大：流行的开源项目每天会收到大量的报告。例如，Eclipse平台在2004-2005年间每天平均收到37份，最高达220份；Mozilla项目在2005年每天有将近300个报告需要分类 (triaging)。手动处理这些报告是不现实的。
  2. 难以理解：特别是涉及性能（如“运行缓慢 slow”）的缺陷，缺乏精确的预言 (oracle)，经验丰富的工程师也需要数周时间才能理解少数报告。

面临的挑战 (Why is it Challenging?)

  1. 自然语言的模糊性：缺陷报告主要用自然语言和代码片段表达，自然语言浩瀚且模棱两可 (vast and ambiguous)。
  2. 规模巨大：大型复杂软件中的问题报告数量极其庞大 (huge)。
  3. 表达质量差 (Poorly expressed)：存在信息缺失、文本嘈杂的缺陷报告，这带来了额外的技术挑战（例如只有一句“it does not work”，需要大量的来回沟通才能明确问题）。

缺陷报告的生命周期 (Lifecycle of A Bug Report)

理解缺陷报告的生命周期有助于我们知道AI可以在哪些阶段发挥作用：

  • UNCONFIRMED -> NEW: 未确认到新报告。
  • NEW -> ASSIGNED: 新报告分配给开发人员。
  • ASSIGNED -> RESOLVED: 开发人员接手并解决（状态可能为 FIXED, WONTFIX, WORKSFORME, DUPLICATE, INVALID）。
  • RESOLVED -> VERIFIED: 解决方案被验证。
  • VERIFIED -> CLOSED: 关闭。
  • 在某些情况下，Bug可能会被重新打开 (REOPEN)。

AI 在缺陷报告分析中的应用 (AI for Bug Report Analysis)

AI技术可以在缺陷报告生命周期的不同阶段提供帮助：

  1. 缺陷报告优化 (Bug report optimization) (从 unresolved 到 new):
    • subnote_Bug_Report_Classification: 将真正的缺陷报告与功能增强或重构请求区分开来。
  2. 缺陷报告分类/分流 (Bug report triage) (从 new 到 assigned):
    • subnote_Bug_Report_Prioritization: 预测缺陷报告的优先级 (Priority)。
    • subnote_Duplicate_Bug_Report_Detection: 检测重复的缺陷报告 (Duplicate)。
  3. 缺陷修复 (Bug fixing) (从 assigned 到 resolved):
    • subnote_Bug_Localization: 使用缺陷报告来定位代码中的缺陷位置 (Localize bugs)。
