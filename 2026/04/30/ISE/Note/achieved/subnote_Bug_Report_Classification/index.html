Bug Report Classification (缺陷报告分类)

Bug Report Classification (缺陷报告分类)

缺陷报告分类发生在其生命周期的早期阶段（例如从未解决转移到新建状态）。在GitHub等平台上，用户提交的“Issue”并不一定都是Bug，很多可能是功能请求 (feature requests) 或一般性问题。

目的与特征

  • 目的 (Purpose): 区分关于缺陷 (bugs)（即纠正性维护）的报告与关于其他活动的报告（如功能增强、重构），或将缺陷与非缺陷 (non-bug) 报告区分开来。
  • 特征 (Feature): 报告的标题 (title)、描述 (description) 和评论/讨论 (comments/discussion)。
  • 原因 (Why?): 帮助加速维护过程，节省分析人员的精力。

传统机器学习方法 (Traditional ML Approach)

早期工作 (如 Antoniol et al. 2008) 使用传统 NLP 和机器学习模型（如决策树 Decision Tree、朴素贝叶斯 Naive Bayes、逻辑回归 Logistic Regression）。主要步骤包括：

  1. 文本过滤 (Text filtering):
    • 去除标点符号。
    • 特定转换：拆分路径、电子邮件和驼峰式标识符 (camel-case identifiers)（例如将 “MyUser account” 拆分为 “My”, “User”, “account”）。
    • 注意：通常不去除像 “should”, “might”, “not” 这样的停用词 (stop words)。因为 “This is not a bug” 如果去除了 “not”，语义会完全颠倒，严重影响分类。
  2. 词干提取 (Stemming):
    • 通过规则将单词的不同形式还原为其共同词根（例如：caresses -> caress, ponies -> poni）。
  3. 索引 (Indexing): 将文本转换为数字向量。
    • 词频 (Term Frequency, TF): 一个词在报告中出现的频率。
    • 传统NLP通常使用 TF-IDF (词频-逆文档频率)。但在缺陷分类中，通常不使用 IDF，只使用 TF。因为像 “failure”, “crash”, “should” 这样的词虽然在许多文档中频繁出现，但它们构成了非常重要的指示信息，不应被 IDF 惩罚。
  4. 训练与预测:
    • 将文本表示为向量后，赋予标签（如 {bug, no bug} = <0, 1>），然后输入到分类器（DT, NB, LR）中进行训练和预测。

基于深度学习与语义表示的方法 (Deep Learning & Learned Representation)

传统的 TF 方法丢失了单词的语义信息。后来的工作采用了深度学习（如 Transformer, BERT）。

缺陷报告的半结构化特性 (Semi-structured nature)

目前的许多工具未能处理缺陷报告的半结构化特性。报告不仅包含自然语言，还包含代码片段、日志等，且各部分的信息重要性不同。

多方面分层注意力网络 (MHNurf)

这是一种结合了语义学习 (semantic learning) 和特征工程 (feature engineering) 的方法，用于识别与性能缺陷相关的报告。

它基于三个关键知识点：

  1. 语义很重要 (Semantics matter): 同样的单词或代码标记在不同的上下文中含义不同。
  2. 存在多种特征类型: 除了标题和描述，累积的评论 (accumulated comments) 等也提供有用信息。
  3. 并非所有部分都同样相关: 报告中只有少数句子或单词在识别性能缺陷时是关键的。

技术细节:

  • 使用 GloVe 进行词嵌入 (word embedding)，无监督学习将词转化为包含语义的向量。
  • 采用双层注意力机制 (Two levels of attentions)：词/标记级别 (word/token level) 和 句子/语句级别 (sentence/statement levels)。
  • 可以灵活地组合不同特征类型（标题、描述、评论、代码）进行并行语义学习和拼接 (concatenation)。
  • 具有跨项目 (cross-project) 应用的泛化能力。
