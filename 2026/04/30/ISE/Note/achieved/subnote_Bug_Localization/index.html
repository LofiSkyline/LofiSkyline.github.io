Bug Localization using Report (利用报告进行缺陷定位)

Bug Localization using Report (利用报告进行缺陷定位)

  • 目的 (Purpose): 一旦识别出缺陷，开发人员应定位包含该缺陷的源代码文件 (source code files)（或方法、类等）。寻找缺陷位置的过程称为缺陷定位。
  • 原因 (Why?): 减少缺陷修复时间和维护成本。

注意：虽然有很多工作侧重于仅使用源代码进行缺陷定位，但这里的方法是结合缺陷报告的信息。

  • 目标 (Goal): 给定一个新缺陷报告（确认包含缺陷）和缺陷跟踪系统，定位缺陷所在。

学习排序方法 (Learning to Rank Approach)

Ye 等人 (2014) 提出了一种利用“学习排序 (learning to rank)”技术的经典工作。该方法包含全面的特征工程（7个特征），学习一个排序模型，输出与缺陷报告最相似的源代码文件排名。

特征工程亮点

  1. TF-IDF 余弦相似度 (Feature 1):

    • 使用 TF-IDF 将缺陷报告 ® 和源代码文件 (s) 转换为向量，计算它们的余弦相似度 (Cosine similarity)。
    • 还计算表面词汇相似度 (Surface lexical similarity)。
    • 局限性：这种基于直接文本匹配的方法仅在源代码包含广泛且全面的注释，或缺陷报告中直接包含代码片段/类名/方法名时才有效。如果报告描述（如“Toolbars missing icons”）与源代码（如实现类 PartRenderingEngine）没有共享词汇，该方法就会失效。

  2. 连接 API 规范 (Feature 2):

    • 为了解决词汇鸿沟问题，解决方案是通过 API 规范 (API specification) 更好地连接源代码和报告。提取源文件中所有方法的 API 规范并连接成一个整体文档，计算报告与 API 文档的相似度。

  3. 协同过滤效应 (Feature 3 - Collaborative filtering effect):

    • 以前修复过某缺陷的文件，可能也是导致类似新缺陷的易错文件。计算新报告与“该源文件曾被修复过的所有历史缺陷报告”之间的相似度。

  4. 类名长度 (Feature 4):

    • 当源文件的类名出现在报告中，并且类名越长 (longer)（因此越具体 specific），相似性信号就越强。

  5. 近期修复记录 (Feature 5):

    • 最近 (recently) 被修复过的源代码文件比很久以前被修复过、或从未被修复过的文件更有可能仍然包含缺陷。

  6. 修复频率 (Feature 6):

    • 被频繁 (frequently) 修复的源文件可能是易错文件 (fault-prone file)。计算源代码被修复的次数。

排序与输出

  • 模型通过优化程序（定义损失函数 loss function）来寻找一组权重参数，使得已知与缺陷报告相关的源文件能在评分函数中排名靠前 (top of the list)。
  • 在测试/使用阶段，给定任意缺陷报告，模型计算项目中每个源文件的得分，并降序排列。
  • 用户（开发者）会看到一个排序后的文件列表 (ranked list of files)。期望是排名靠前的文件包含缺陷原因的可能性更高。
