Classic Metrics for Defect Prediction (经典缺陷预测度量标准)

软件缺陷预测始于指标识别 (metrics identification)，这也是80年代的主要研究方向。核心假设是：代码越复杂（拥有更多的路径或更多的操作符），引入缺陷的可能性就越大。

1. McCabe 圈复杂度 (McCabe Cyclomatic Complexity)

定义: 基于代码控制流图中线性独立路径的数量来衡量代码的复杂性。代码中的条件语句 (condition statements) 越多，可能的执行路径就越多，圈复杂度就越高。

计算方法

  • 图论公式:
    $$ \text{Cyclomatic Complexity} = E - N + 2 $$
    （其中 $E$ 是控制流图的边数，$N$ 是节点数）。
  • 简单计算法则:
    $$ \text{McCabe cyclomatic complexity} = \text{number of simple conditions} + 1 $$
    其中，简单条件 (simple conditions) 是指不包含 OR 或 AND 的条件语句。
    • if (a > b) 算作 1 个简单条件。
    • while (a > b) 算作 1 个简单条件。
    • for (a=b; a > b; b++) 算作 1 个简单条件。
    • do... while (a > b) 算作 1 个简单条件。
    • case 语句中的每一个分支 (不包括 default) 算作 1 个简单条件。
    • catch 异常处理块算作 1 个简单条件。
  • 复合条件 (Compound conditions): 对于包含 AND 或 OR 的条件，需要计算其内部的简单条件数量。
    • if (a > b) OR (a > 2) 算作 2个 简单条件。
    • if (a > b && a > 1) 算作 2个 简单条件。

注意边界: 在计算圈复杂度时，不要深入外部类/库的函数内部 (We do not go into the function of external classes/libraries)，只需计算当前可见代码块中的条件数量。即使代码中没有任何条件分支（顺序执行），其圈复杂度也是 $0 + 1 = 1$。

2. Halstead 复杂度度量 (Halstead Complexity Measures)

定义: 基于代码中使用的运算符 (operators) 和操作数 (operands) 的数量来衡量代码复杂性。

  • $n_1$ = 独立运算符的数量 (E.g., !=, !, %, /, *, +, &&, ||, 等)
  • $n_2$ = 独立操作数的数量 (E.g., 非保留字的标识符、常量、类型如 bool, char, double)
  • $N_1$ = 运算符出现的总次数
  • $N_2$ = 操作数出现的总次数

基于上述基本计数，可以计算出：

  • 代码词汇量 (Code vocabulary): $n = n_1 + n_2$
  • 代码长度 (Code length): $N = N_1 + N_2$
  • 代码体积 (Volume): $V = N \log_2 n$
  • 难度 (Difficulty): $D = \frac{n_1}{2} \times \frac{N_2}{n_2}$

3. 代码行数与注释行数 (Lines of Code and Comment)

  • 代码行数 (LOC): 包含代码的行数。一般来说，代码段越大，包含缺陷的可能性就越大。
  • 注释行数 (Lines of Comment): 包含注释的行数。通常预期注释越多的代码段包含的缺陷越少（注释代表代码逻辑可能经过了更仔细的思考和说明）。

经典度量的局限性

虽然上述指标很直观，但我们不知道某个指标到底使得缺陷发生的可能性增加了多少，也不知道这些指标之间是如何相互作用的 (how each of these metrics interacts with other metrics)。因此，我们需要使用机器学习模型，将这些指标作为特征 (features) 输入，来“学习”它们与缺陷之间的隐藏关联。
