Topic 08: Clocks

Topic 08: Clocks

在分布式系统中，时间（Time）不仅是一个记录单位，更是协议运行和系统行为建模的基础。本笔记将深入探讨从物理时钟同步到逻辑因果关系的演进。

1. The Importance of Time (时间的作用)

In a distributed system, time provides an abstract, monotonically increasing function that allows us to reason about the system’s state and behavior. Its key roles include:

  • Ordering & Sequencing: Determining the sequence of events.
  • Causality (因果关系): Capturing whether event $A$ could have influenced event $B$.
  • Synchronising Events: Ensuring nodes take action at the “same” moment.
  • Mutual Exclusion: Using timestamps as priority to resolve resource contention.
  • Time-stamping Sensor Values: Aligning data from different sources in safety-critical systems (e.g., nuclear plants).
  • Scheduling: Managing task execution and timeouts across nodes.

2. Physical Clocks: Concepts and Definitions

2.1 Clock Function

A local physical clock is modeled as a mapping from Real Time ($t$, also called Newtonian timeframe) to Local Clock Time ($T$):
$$C: R \rightarrow L$$

  • Perfect Clock: A clock where $C(t) = t$ for all $t$. Every unit of real time results in exactly one unit of local time.

2.2 Drift Rate and Skew

  • Drift Rate ($\rho$, rho): The rate at which a local clock deviates from real time.
    • In reality, physical clocks are based on Crystal Oscillators (石英晶体振荡器). Even the same batch of crystals will oscillate at slightly different frequencies.
    • Fast Clock: $\frac{dC}{dt} > 1$.
    • Slow Clock: $\frac{dC}{dt} < 1$.
    • Drift Formula: $1 - \rho < \frac{C(t_2) - C(t_1)}{t_2 - t_1} < 1 + \rho$.
  • Skew ($\delta$, delta): The difference between the readings of two different clocks at the same real time $t$:
    $$\text{Skew} = |C_1(t) - C_2(t)|$$

2.3 Clock States

  • Non-faulty Clock: A clock is considered non-faulty over a time interval if it is monotonic, differentiable, and its drift rate is bounded by $\rho$.
  • $\delta$-synchronised: Two clocks $C_1$ and $C_2$ are $\delta$-synchronised at time $t$ if $|C_1(t) - C_2(t)| \le \delta$.
  • Well-synchronised (良好同步): A set of clocks is well-synchronised if any two non-faulty nodes are $\delta$-synchronised.
    • Note: You must define “non-faulty” and “$\delta$-synchronised” to properly define “well-synchronised”.

3. Real-Time Synchronisation (RTS)

A Reliable Time Source (RTS) is a base reference that distributes “correct” time (e.g., UTC).

3.1 Requirements for RTS Algorithms

  1. Frequency: Time must be distributed frequently enough to bound the skew.
  2. Smoothness: No node should perform too large an adjustment in a single action (e.g., clocks must not go backwards or “zoom” forwards).

3.2 Cristian’s Algorithm

  • Mechanism: A client periodically requests time from a Time Server (connected to a source like WWV broadcasting UTC).
  • Problems:
    • Latency: The time received is already old due to network delay.
    • Jumping: Direct setting violates smoothness.
  • Refinement:
    • Calculate round-trip time ($RTT$) and estimate latency: $T_{new} = T_{server} + \frac{RTT}{2}$.
    • Introduce corrections slowly (Gradual Adjustment) to avoid backward steps.

3.3 The Berkeley Algorithm

Used when no external UTC source is available. It focuses on internal agreement.

  1. Polling: A chosen Time Server polls all clients for their current times.
  2. Computing: The server computes a Fault-tolerant Average of the readings (discarding outliers).
  3. Adjusting: The server sends each client an offset (an amount to add or subtract) rather than an absolute time.
  4. Gradual Adjustment (FIX Function):
    • Clients apply the offset gradually over an Adjustment Interval (AI).
    • Goal: AI must be long enough to preserve Accuracy (not changing the rate too much) but short enough to maintain Agreement (keeping nodes close).

4. General Clock Synchronisation Protocol

4.1 Parameters and Event Detection

Nodes need to know when to trigger a resynchronisation event. This can be based on their own Virtual Clock (VC).

  • $R$: The predefined interval on the virtual clock between resynchronisation events.
  • $r_{min} = \frac{R}{1 + \hat{\rho}}$: The minimum real time elapsed between two events.
  • $r_{max} = \frac{R}{1 - \hat{\rho}}$: The maximum real time elapsed between two events.
  • $\beta = \frac{\hat{\delta}}{1 - \hat{\rho}}$: The maximum real-time difference (skew) between nodes reaching the resynchronisation point.

4.2 Pseudo-code Flow

The protocol follows three core steps:

  1. Event Generation: Triggered when $VC_p$ reaches $jR$.
  2. Reading Clock Times: Each node reads/estimates the clock values of other nodes.
  3. Applying Correction: Use a Convergence Function to calculate the new virtual clock state.

4.3 Convergence Functions

A convergence function takes $n+1$ inputs and must satisfy:

  • Monotonicity: Increasing an input increases the output.
  • Translation Invariance: $f(x_1+S, …, x_n+S) = f(x_1, …, x_n) + S$.
  • Accuracy Preservation: If all inputs are within a range, the output stays within a related range.

Common Functions:

  • Egocentric Average: Average of others and self.
  • Fault-tolerant Midpoint: Discard $k$ highest and $k$ lowest, take midpoint of remaining.
  • Fault-tolerant Average: Discard $k$ highest and $k$ lowest, take average of remaining.

5. Logical Clocks (逻辑时钟)

In asynchronous systems, physical time is hard to maintain. Lamport suggested focusing on Event Ordering instead.

5.1 Happened-before Relation

Defined by three rules:

  1. Same Process: If $a$ occurs before $b$ in process $P_i$, then $a \to b$.
  2. Message Passing: If $a$ is the sending of a message and $b$ is the receipt of that message, then $a \to b$.
  3. Transitivity: If $a \to b$ and $b \to c$, then $a \to c$.
  • Concurrency ($\parallel$): If neither $a \to b$ nor $b \to a$, the events are concurrent.

5.2 Lamport Logical Clock Rules

Each process $P_i$ maintains a local counter $L_i$.

  • IC1 (Internal): Before each event at $P_i$, increment $L_i$: $L_i = L_i + 1$.
  • IC2 (Message):
    • When $P_i$ sends a message $m$, it attaches $t = L_i$.
    • When $P_j$ receives $(m, t)$, it updates: $L_j = \max(L_j, t) + 1$.

5.3 Limitations of Logical Clocks

Clock Condition: $a \to b \Rightarrow L(a) < L(b)$.
The Flaw: The reverse is NOT true. $L(a) < L(b)$ does not imply $a \to b$.

Example:
Process $P_1$ performs an internal event $a$ ($L=1$).
Process $P_2$ performs an internal event $b$ ($L=2$).
Here $L(a) < L(b)$, but $a$ and $b$ are concurrent ($a \parallel b$). We cannot tell if they are causally related just by looking at the numbers.

6. Vector Clocks (向量时钟)

Vector clocks address the limitation of logical clocks by capturing full causal history.

6.1 Vector Clock Rules

Each process $P_i$ maintains a vector $V_i$ of length $n$ (number of processes).

  • IC1 (Internal): Before an event, $P_i$ increments its own component: $V_i[i] = V_i[i] + 1$.
  • IC2 (Message):
    • Send: Attach $V_i$ to the message.
    • Receive: For each $k$, $V_j[k] = \max(V_j[k], V_{msg}[k])$, then increment $V_j[j]$.

6.2 The Strong Clock Condition

$a \to b \iff V(a) < V(b)$

  • $V(a) < V(b)$ means for all $k, V(a)[k] \le V(b)[k]$ AND there exists at least one $j$ such that $V(a)[j] < V(b)[j]$.

Example Solution:
$P_1$ event $a$: $V(a) = [1, 0]$.
$P_2$ event $b$: $V(b) = [0, 1]$.
$V(a)$ is not less than $V(b)$, and $V(b)$ is not less than $V(a)$. Thus, we can correctly conclude $a \parallel b$.

If $P_1$ sends $a$ to $P_2$, and $P_2$ receives it as $c$:
$P_1$ event $a$: $V(a) = [1, 0]$.
$P_2$ receive $c$: $V© = [\max(0, 1), \max(1, 0) + 1] = [1, 2]$.
Now $V(a) < V©$, confirming $a \to c$.
