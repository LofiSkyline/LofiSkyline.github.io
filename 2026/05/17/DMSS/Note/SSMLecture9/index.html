SSMLecture 9: Cloud Data Security

SSMLecture 9: Cloud Data Security

1. 云的分类与动机 (Cloud Classification & Motivation)

云的分类 (Types of Cloud)

  • Public cloud (公有云): Anyone can share the cloud resource with you. 资源对公众开放，由第三方提供商拥有和管理。
  • Private cloud (私有云): The cloud is owned/maintained by your own organization. 专为单一组织内部使用。
  • Community cloud (社区云): A group gets together to jointly own/manage a cloud infrastructure. 由多个有共同利益（如安全需求、政策）的组织共同拥有和管理。
  • Hybrid cloud (混合云): Some mix of public/private. 例如：使用私有云处理核心数据，公有云进行海量存储。

使用动机 (Motivation)

云服务的核心吸引力在于：Simplicity (简单性)、Cost (低成本)、Security (安全性)、Resilience (弹性)、Flexibility (灵活性) 以及 Pace of Innovation (创新速度)。

2. 云安全的主要问题 (Main Cloud Security Issues)

根据 Cloud Security Alliance (CSA) 的总结，云环境面临 12 个主要的威胁：

  1. Data breaches (数据泄露): Sensitive information is released, viewed, stolen or used by an unauthorized individual.
    • Case: 2018 年 Facebook 因 “View As” 漏洞导致 5000 万账户被入侵；2017 年 Equifax 泄露 1.43 亿美国人的社保号。
  2. Insufficient identity, credential and access management (身份与访问管理不足): 缺乏多因素认证 (MFA) 和密钥轮换机制。
    • Case: 开发者将凭证上传至 GitHub，攻击者 36 小时内即可扫描并劫持账户用于挖矿。
  3. Insecure interfaces and APIs (不安全的接口和 API): 云服务的安全高度依赖基础 API 的安全性。
    • Case: 2015 年美国国税局 (IRS) “Get Transcript API” 漏洞暴露 30 万条记录。
  4. System vulnerabilities (系统漏洞): 利用内核或系统库中的 bug。
    • Case: Heartbleed (OpenSSL) 导致密钥可被读取；Shellshock (Bash) 允许执行任意命令。
  5. Account hijacking (账户劫持): 攻击者通过网络钓鱼或漏洞获取用户凭证。
    • Case: Code Spaces 因 AWS 账户被劫持且未开启 MFA，导致公司资产全毁而倒闭。
  6. Malicious insiders (恶意内部人员): 现任/前任员工滥用授权权限。
    • Case: Google 系统工程师滥用权限访问未成年人隐私数据。
  7. Advanced Persistent Threats (APTs) (高级持续性威胁): 建立长期立足点并横向移动。
    • Case: Carbanak 团伙瞄准银行内部系统；Stuxnet 蠕虫攻击 PLC。
  8. Data loss (数据丢失): 因意外删除、物理灾难或丢失加密密钥 (Encryption Key) 导致。
    • Case: 2011 年 Amazon EC2 崩溃导致显著的数据丢失。
  9. Insufficient due diligence (尽职调查不足): 盲目采用云服务商。
    • Case: Nirvanix 突然倒闭，仅给客户两周时间迁移海量数据。
  10. Abuse and nefarious use of cloud services (滥用云服务): 攻击者注册免费/被盗账户发起攻击。
    • Case: MIRAI 僵尸网络 生成超过 300 Gbps 的流量。
  11. Denial of service (DoS) (拒绝服务攻击): 耗尽系统资源（带宽、CPU）使服务不可用。
  12. Shared technology issues (共享技术问题): 多租户架构隔离不足。
    • Case: Cross-VM Side Channels 侧信道提取私钥；VENOM 漏洞允许从访客机逃逸。

3. 攻击者模型 (Attacker Models)

攻击者模型描述了攻击者在尝试破坏系统时具备的能力和权限：

  1. Honest (诚实模型): 严格遵守规则，不构成威胁。
  2. Honest-but-curious (诚实但好奇): 云安全中最常用的模型。
    • 特点：遵守协议 (Obey protocol)，但作为 Passive attacker (被动攻击者)，会尝试访问数据。
  3. Malicious-but-cautious (恶意但谨慎):
    • 特点：不一定遵守协议，但会竭力保持隐蔽 (Remain undetected)，不留下可验证证据。
    • 为什么 CSP 符合此模型？
      • Malicious: 只要能通过作弊获取额外利益（如数据变现），CSP 就有动机。
      • Cautious: 为了维持商业信誉和留住客户，它不能公然破坏系统或被发现。它是一个 Active and covert attacker (主动且隐蔽的攻击者)。
  4. Malicious (恶意模型): 最强大的攻击者。
    • 特点：对网络和资源拥有完全控制权 (Complete control)，不择手段，不在乎是否被发现（如 Dolev-Yao 模型）。

4. 案例分析 1: CryptDB (加密数据库系统)

架构关系 (Architecture)

CryptDB 作为一个安全中间件 (Middleware)，其核心架构包括：

  • User / Application Server: 发送明文 SQL。
  • CryptDB Proxy: 位于 Trusted Zone。负责存储密钥、拦截 SQL、将查询重写为加密 SQL，并解密 MySQL 返回的结果。
  • DBMS Server (MySQL): 存储加密数据，运行 CryptDB UDFs。它是 Unmodified (未经修改的)，只需安装 UDF 插件。

洋葱加密层级 (Onion Encryption Layers)

CryptDB 为每列数据维护多个独立的“洋葱”：

  1. Onion Eq (等值查询洋葱): RND -> DET -> JOIN
  2. Onion Ord (排序查询洋葱): RND -> OPE -> OPE-JOIN
  • RND (Random): 最强加密，每次加密结果不同，不支持任何运算 (No functionality)。
  • DET (Deterministic): 确定性加密。相同的明文对应相同的密文，支持 Equality selection (等值查询)，但泄露频率分布。
  • OPE (Order-preserving): 保序加密。如果 $x < y$，则 $enc(x) < enc(y)$。支持 Sort (排序) 和范围查询，但泄露数据顺序。

UDF 与 剥洋葱机制 (UDF & Onion Peeling)

  • 何时调用: 当用户的查询需要某个被包裹在 RND 内部的层级（如 DET 或 OPE）时。
  • 过程:
    1. Proxy 将 解密密钥 (Decryption Key) 发送给数据库端的 UDF。
    2. UDF (User Defined Function) 在 MySQL 内部执行，将数据从强加密层“脱”下，露出里面的层级。
    3. 剥洋葱后，数据会永久保留在较低的安全级别，因为重加密的性能开销极大且会影响后续查询效率。这就是为什么“降级后无法恢复”的原因。

用户查询完整流程与降级机制 (Query Workflow & Downgrade)

以查询 Age 列为例，系统会根据当前的加密状态 (State) 决定是否降级：

  1. 不降级情况 (No Downgrade):
    • 如果用户执行 SELECT * FROM Employees WHERE Age = 30，且 Age 列的 Onion Eq 已经处于 DET 级别（可能因为之前的查询已经剥过皮了）。
    • Proxy 直接将 30 加密为 DET_30，MySQL 直接执行密文比对，不调用 UDF。
  2. 触发降级 (Peeling Trigger):
    • 等值比对 (Equality Selection): 如果 Age 为 RND 状态，Proxy 发送 DET 密钥，调用 UDF 剥开 RND 层，降级至 DET。
    • 排序/范围比对 (Sort / Range): 如果执行 ORDER BY Age，Proxy 发送 OPE 密钥，调用 UDF 剥开 Onion Ord 的 RND 层，降级至 OPE。
  3. MySQL 原生引擎 vs UDF:
    • 一旦降级到 DET 或 OPE，MySQL 使用其原生引擎 (Native Engine) 和索引（如 B-Tree）即可完成搜索，效率极高。
    • UDF 仅在“剥皮”瞬间和执行“同态加法 (HOM)”等复杂运算时被调用。

为什么降级后无法恢复？ (Why Restoration is Impossible)

  • Performance (性能): 重加密 (Re-encrypting) 需要对全表数百万行记录进行 CPU 密集型的 RND 加密操作，开销巨大。
  • Query Frequency (查询频率): 统计表明一旦某列被查询过，近期极大概率会被再次查询。频繁的“穿脱衣服”会导致系统可用性 (Availability) 降至零。
  • Result: 这是一个 Security-Efficiency Tradeoff（安全与效率的权衡）。系统倾向于保持较低的安全级别以换取高性能。

5. 案例分析 2: ConfiChair (SaaS 应用管理)

三个角色的意图 (Roles & Intentions)

  • Author (作者): Intent to submit papers and scores without letting the Cloud see them.
  • Chair (主席): Intent to manage the conference. 作为 Trusted Third Party (可信第三方)，持有核心密钥。
  • Reviewer (审稿人): Intent to evaluate papers assigned by the Chair.

“主席重加密”过程 (Re-encryption Logic)

为了瞒过 CSP，ConfiChair 使用了以下逻辑：

  1. Author Upload: 作者用论文密钥 $k$ 加密论文，再用会议公钥 $pub(Conf)$ 加密 $k$，得到密钥块 ${k}_{pub(Conf)}$。
  2. Chair Intervenes: 主席下载 ${k}_{pub(Conf)}$，用会议私钥 $priv(Conf)$ 解密出明文 $k$。注意：这意味着主席能看到论文内容，但 CSP 依然看不到。
  3. Chair Re-encrypts: 主席用审稿人已知的对称密钥 $K_{Conf}$ 重新加密 $k$，得到 ${k}{K{Conf}}$ 并传回云端。
  4. Reviewer Access: 审稿人下载新块，用 $K_{Conf}$ 拿回 $k$，最终读取论文。

安全属性与局限性

  • Secrecy (机密性): 系统不知道论文、评分、排名、讨论的具体内容。
  • Unlinkability (不可链接性): 理想情况下，系统不应知道哪个审稿人 R 评审了哪个作者 A 的论文。
  • 现实的代价 (The Cost):
    • Traffic Analysis (流量分析): CSP 虽然打不开保险箱，但可以“观察邮差”。如果 Alice 寄出一个 5MB 的包，Richard 随后取走一个 5MB 的包，CSP 就能推断出他们的对应关系。
    • 结论: ConfiChair 成功瞒过了 CSP 的眼睛（内容安全），但无法隐藏整个通信的规程架构和流量特征。
