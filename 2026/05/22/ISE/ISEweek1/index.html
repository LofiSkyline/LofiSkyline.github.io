课件的第 12 到 19 页详细讲解了如何利用 AI 和传统机器学习（Traditional Machine Learning） 算法来进行 Bug Report Classification（缺陷报告分类）。
它的核心目的非常明确：当用户提交一个文本报告时，AI 要自动判断这到底是一个真正的缺陷（Bug/Corrective Maintenance），还是一个功能增强需求（Enhancement/Refactoring）。
这一部分的内容可以分为两大核心：数据预处理（Data Pre-processing） 和 分类模型（Classification Models）。

一、 数据预处理的三个步骤 (Data Pre-processing)

在把一封自然语言写成的 Bug 报告喂给 AI 之前，必须先把它数字化。Antoniol 等人在 2008 年的早期经典工作奠定了这个流程：

1. 文本过滤 (Text Filtering)

  • 标点符号移除（Punctuation removal）： 删掉无用的标点。
  • 特定转换（Specific transformations）： 比如拆分路径、邮箱地址和驼峰命名（Camel-case identifiers）。例如，把 MyUser account 拆分为 My、User 和 account。
  • ⚠️ 核心考点：为什么不移除停用词（Stop Words）？
    在常规 NLP 中，我们会去掉 “not”、“should” 这种高频词。但缺陷报告中绝对不能去掉！课件给出了经典反例：如果把 This is not a bug 中的停用词去掉，文本就会变成 This is a bug，这会导致严重的误判。

2. 词干提取 (Stemming)

  • 概念： 将一个单词的不同时态、单复数等形式，还原为它的基本词干（Common Base）。
  • 规则举例（课件第 14 页）： 比如末尾有 SSES 替换为 SS（caresses \rightarrow caress），或者去掉末尾的 S（cats \rightarrow cat）。
  • 效果对比： * Original: light caresses colours, sets them aglow…
    • After Stemming: Light caress colour, set them aglow…

3. 特征索引 (Indexing)

处理好文本后，需要将其转化为由数字组成的向量（Vector）。

  • 词频 (Term Frequency, TF)： 计算一个词在当前报告中出现的比例。比如一封 150 词的报告中，bug 出现了 50 次，则 tf = 50/150 = 0.3333。
  • ⚠️ 核心考点：为什么在这里往往不用 TF-IDF，而只用 TF？
    通常的文本分类会用 IDF 来惩罚在所有文档中都频繁出现的词。但在缺陷报告中，像 failure（失败）、crash（崩溃）或 should（应该）这类词虽然在绝大多数报告里都出现，但它们包含了极度重要的特征信息，不能被 IDF 惩罚（Penalized）。
  • 最终产物： 每一封 Bug 报告都变成了一个数值向量，标签（Label）则是二分类向量，如 {bug, no bug} = <0, 1>。

二、 三种经典的 AI 分类模型 (Classification Models)

数据变成数字向量后，就可以训练以下三种经典的机器学习模型了：

1. 决策树 (Decision Tree / DT)

  • 工作原理： 它是一种树状结构，根据属性的不同取值对数据进行一层层的切分（Split）。每个非叶子节点都是一个属性测试，分支代表测试结果。
  • 直观理解： 课件第 17 页举了一个“判断一个人是否健康（Fit）”的例子。先看 Age < 30?，如果是，再看 Eat’s a lot of pizzas?；如果不是，再看 Exercises in the morning?，最终走到叶子节点输出 Fit 或 Unfit。在 Bug 分类中，节点会变成“是否包含 crash 词汇？”等文本特征。

2. 朴素贝叶斯 (Naive Bayes / NB)

  • 工作原理： 基于**贝叶斯定理（Bayes theorem）**的条件概率分类器。
  • 数学公式：
  • 逻辑： 它的核心思想是计算给定特征 x 下，该报告属于各个类别 C_k（是 Bug 或 不是 Bug）的条件概率，哪个概率高就分到哪一类。

3. 逻辑回归 (Logistic Regression / LR)

  • 工作原理： 虽然名字叫回归，但它是一个用于分类的线性模型。它将自变量 x_1, x_2, …, x_N 进行线性组合得到 y’，然后通过 Sigmoid（S型）激活函数 映射到 [0, 1] 空间，输出一个概率值。

总结与承上启下

这几页介绍的是 2008 年左右最经典的“文本挖掘 + 传统机器学习”方案。它的优点是简单、计算快，但缺点也很明显：它把文本打碎成了独立的词（Bag of Words），完全丢失了词序和上下文语境。
这就引出了第 20 页之后的内容——现代研究如何利用 深度学习（Deep Learning, 如 MHANurf） 来解决这些传统方法的缺陷。
关于这三大传统模型的预处理和推导，你有什么地方需要我们用实例推演一下吗？
为了应对考试中 “简要描述其工作原理（Briefly understand how they work）” 的要求，你需要掌握这三个经典模型最核心的工作机制（Mechanism）和独特性质（Characteristics）。
在答题时，用英文专业词汇（Keywords）精准命中得分点至关重要。以下是为你整理的冲刺版背诵框架：

1. Decision Tree (DT - 决策树)

  • 工作方式 (How it works): 它是一种树状结构 (Tree structure)。模型通过在每个节点上对特征属性进行测试/切分 (Split/Test)，根据不同的特征值将数据分流到不同的分支 (Branch)，层层向下流转，直到最终的叶子节点（Leaf node）输出分类标签（如 Bug 或 非 Bug）。
  • 特性 (Characteristics):
    • 直观易懂 (Highly interpretable): 它的运行逻辑非常符合人类的条件判断思考方式（If-Then 逻辑）。
    • 非线性 (Non-linear structure): 能够天然地处理复杂的非线性关系。

2. Naive Bayes (NB - 朴素贝叶斯)

  • 工作方式 (How it works): 它是基于贝叶斯定理 (Bayes theorem) 的概率分类器。模型会计算在给定的文本特征 x 下，该报告属于每一个类别 C_k 的条件概率 (Conditional probability)。计算完所有类别的概率后，选择概率最高 (Higher probability) 的那一类作为预测结果。
  • 特性 (Characteristics):
    • “朴素”假设 (Naive assumption): 它有一个强假设，即假设所有的文本特征（即词汇之间）是相互独立 (Feature independence) 的。虽然这个假设在现实中往往不成立，但该模型在文本分类任务中却异常高效。
    • 计算快 (Computationally efficient): 只需要进行简单的计数和概率乘法，训练和预测速度极快。

3. Logistic Regression (LR - 逻辑回归)

  • 工作方式 (How it works): 它是一个用于分类的线性模型 (Linear model)。它首先将输入的自变量 x 进行线性组合，然后通过一个 Sigmoid 激活函数 (Sigmoid / Logistic function) 将结果映射为一个 [0, 1] 之间的概率值 p(x)。最后利用一个截断点/阈值 (Cutoff point) 来决定最终归属于哪一类。
  • 特性 (Characteristics):
    • 输出概率值 (Outputs probabilities): 它不仅给出分类，还能给出模型对这个分类的信心程度（即概率值）。
    • 易于优化 (Easy to train): 它是统计学中的经典方法，对线性可分的数据集表现极佳。

💡 考试答题套路提示

如果考题要求对比它们，记住一句话：

DT 靠“层层规则切分”，NB 靠“算条件概率”，而 LR 靠“线性映射加 Sigmoid 函数”。它们在 Bug 分类中，都是为了处理由文本词频（TF）构成的特征向量。

好的，我们来详细拆解第 9 页提到的 Lifecycle of A Bug Report（缺陷报告的生命周期）。
在现代软件工程中，Bug 报告并不是一个静态的记录，而是一个动态流转的状态机（State Machine）。随着开发人员、测试人员和用户的介入，它的状态会不断发生变化。
结合课件中的流程图，整个生命周期可以详细分为以下几个核心阶段和流转逻辑：

1. 初始阶段：从未确认到确认

  • UNCONFIRMED（未确认）
    • 背景： 当 end-users（终端用户）或测试人员刚在 JIRA 或 GitHub 等 Tracker 上提交一个新的缺陷时，系统默认给出的初始状态。
    • 流转： 此时，该报告还没有经过官方的验证。如果分拣人员（Triager）或开发人员复现了这个问题，确认这确实是一个 Bug（Indeed a bug），它就会被推进到下一个状态。
  • NEW（新提交）
    • 背景： 报告已被确认是软件本身的缺陷。
    • 流转： 它会进入待办列表（Backlog），等待被分发。如果在这个阶段发现报告有误，或者负责人发生了变更，它的所有权可能会被修改（Ownership is changed）。

2. 处理阶段：指派与修复

  • ASSIGNED（已指派）
    • 背景： **某位特定的开发人员接管了（takes possession of）**这个 Bug。他的任务就是去阅读 Description、分析 Code、并在本地定位和修复它。
    • 流转： 当开发人员编写完补丁（Patch）并认为**缺陷已经解决（The bug has been solved / Developer has solved the bug）**后，他会将状态修改为 RESOLVED。

3. 解决阶段：五种不同的 Resolution

当 Bug 报告进入 RESOLVED（已解决） 状态时，必须指定一个具体的 Resolution（解决结论）。课件中列出了 5 种最经典的结论：

Resolution 结论 含义 (Meaning)
FIXED 真正修复。代码已被修改，补丁已提交。
WONTFIX 不予修复。可能因为该行为是设计使然（It’s a feature），或者修复成本过高、影响范围太大。
WORKSFORME 无法复现。开发人员在自己的环境下测试发现一切正常，无法重现报告中描述的错误。
DUPLICATE 重复报告。这个 Bug 已经被别人先一步提交了（这正是第 51 页“重复报告检测”要解决的问题）。
INVALID 无效报告。用户描述错误、使用了错误的配置，或者根本不是软件本身的问题。

4. 验证与关闭阶段

  • VERIFIED（已验证）
    • 背景： 缺陷被标记为 RESOLVED 后，不能直接关闭，必须由测试人员（Testers）或项目负责人在特定的构建（Build）中进行回归测试。
    • 流转： 如果测试表明解决方案确实有效（Solution worked），Bug 状态就会推进到 VERIFIED。如果在后续的观察中没有再次收到该缺陷的报告（No occurrence of the bug is reported），它就会被推进到终态——CLOSED（关闭）。

5. 循环流转：重新打开 (REOPEN)

在整个生命周期中，还有一个非常重要的“逆向流转”状态—— REOPEN（重新打开）：

  • 触发条件 1： 当 Bug 被标记为 RESOLVED，但测试人员测试后发现解决方案并不令人满意（Solution is not satisfied），即 Bug 依然存在。
  • 触发条件 2： 当 Bug 已经走到 VERIFIED 甚至 CLOSED 状态，但在新版本中该 Bug 再次复现（Bug is reopened）。
  • 流转： 一旦被 REOPEN，报告就会重新回到 ASSIGNED 或 NEW 状态，由开发人员重新介入排查。

💡 老师敲黑板：为什么我们要学这个？

在后面的课件中，所有的 AI & Data-Driven 技术的目的，都是为了在这些状态切换的“箭头”上提供自动化支持：

  • Bug 报告分类（Classification）： 帮助报告从 UNCONFIRMED 快速优化并分流。
  • 缺陷指派（Triage）与优先级（Prioritization）： 决定从 NEW 到 ASSIGNED 时，该由谁来做、先做哪一个。
  • 缺陷定位（Localization）： 协助开发人员在 ASSIGNED 到 RESOLVED 的修复过程中寻找 buggy files。
