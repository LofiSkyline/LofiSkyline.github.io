Evaluation in Intelligent Software Engineering (智能软件工程中的评估)

在智能软件工程的实验研究中，评价一个算法的好坏不仅仅是看某个指标的数值。由于算法的随机性和现实问题的多目标性，我们需要一套严谨的框架来科学、公平地对比不同的算法或配置（Algorithms / Configurations）。

1. 多目标优化评估 (Multi-objective Evaluation)

许多软件工程任务（如测试用例生成、配置调优）通常涉及多个冲突的目标。例如，在软件测试中，我们希望最大化覆盖率 (Maximize Coverage) 的同时 最小化测试成本 (Minimize Cost)。

核心概念

  • 帕累托前沿 (Pareto Front): 在多目标空间中，不存在任何一个目标在不损害其他目标的情况下得到改进的一组最优解集合。
  • 支配关系 (Domination Relationship): 如果方案 A 在所有目标上都不差于方案 B，且在至少一个目标上优于方案 B，则称 A 支配 B。
  • 帕累托近似 (Pareto Approximation): AI 算法的目标是找到尽可能接近真实帕累托前沿的一组非支配解 (Non-dominated solutions)。

评估指标 (Quality Indicators)

评估两组解集（Solution Sets）的好坏比单目标情况更复杂，通常从以下维度衡量：

  1. 收敛性 (Convergence/Closeness): 衡量解集距离真实帕累托前沿有多近。
    • GD (Generational Distance): 计算解集中每个点到前沿的平均距离。
  2. 多样性 (Diversity/Spread): 衡量解集在空间中分布是否广泛。
    • Spread: 衡量端点覆盖和分布均匀度。
  3. 综合指标: 同时衡量收敛性和多样性。
    • IGD (Inverted Generational Distance): 计算前沿上的点到解集的距离，指标越小越好。
    • HV (Hypervolume): 计算解集与参考点（最差点）之间围成的超体体积。讲师提到：“Hypervolume is usually very widely used to compare solution set quality from two AI algorithms because it naturally prefers well-balanced solutions.”

可视化方法

  • 帕累托散点图 (Pareto Plot): 直观展示两目标间的权衡。
  • 平行坐标图 (Parallel Coordinate Plot, PCP): 适用于 3 个以上目标。横轴为目标编号，纵轴为归一化后的指标值。通过线条的密集程度和形状来判断解集的质量。

2. 统计学对比与假设检验 (Statistical Comparison & Hypothesis Testing)

这是实验设计的核心，旨在回答：观察到的性能差异是由于算法优越性导致的，还是由于随机性导致的？

2.1 统计检验的数据本质：从高维配置到一维观测的“坍缩”

在智能软件工程中，对比的范畴是多层次、跨维度的。我们要对比的往往不是单一算法的微调，而是完全不同的技术路线或数据处理管线 (Pipelines)。统计检验的本质，就是将高维的配置空间映射、坍缩为一维的结果观测值。

深入理解配置对比的三个关键：

  1. 屏蔽内部复杂性 (Abstraction): 不管 Configuration 内部包含了多少组件或超参数，当它们运行后，复杂的结构都会“坍缩”成一个简单的一维数值（如 Accuracy, Latency 或 Coverage）。非参数统计检验（如 Wilcoxon, Friedman）只关注这组一维观测值的相对排名，根本不需要关心背后复杂的模型参数。
  2. 重新定义公平对比 (Fair Comparison): 当我们对比底层原理完全不同的 Config A（跑一轮1小时）和 Config B（跑一轮1秒）时，公平不再仅仅是“迭代次数相同”。这就要求我们限制评估环境和算力开销（如：都在限定 CPU 运行 10 分钟内，看谁的最终一维表现更好）。
  3. 多组检验 (N-Groups Test) 的必然性: 因为不同的降维、聚类和回归方法可以产生几十种排列组合（Configurations），实验极少局限于“两组对比”。这种组合爆炸必然要求我们首先祭出 Kruskal-Wallis 或 Friedman 进行全局多组检验。

2.2 核心概念辨析：Configuration (配置) vs. Seed (随机种子)

初学者常混淆模型的“维度数/簇个数”和“随机种子”。理清两者的数学与实验界限，是正确理解为何要重复运行 30 次并做假设检验的绝对前提：

概念 包含的具体内容（示例） 它的物理本质 它的实验目的
Configuration (配置) 1. 方法选择（如用 PCA 或不降维）
2. 超参数具体数值（如降到 5 维，聚类成 3 个簇）		 系统的设计蓝图。是由工程师或算法可以一行行写死的参数组合。 我们想通过对比不同的配置，找出哪种设计方法和参数组合能带来最好的性能指标。
Seed (随机种子) 诸如 12345、42 等无实际物理意义的数字开关。 随机过程的启动钥匙。决定底层无法控制的随机行为（如 K-Means 初始中心点位置、数据切分策略）。 我们绝对不想优化它。换不同的种子跑 30 次，是为了制造不同的“随机运气环境”，来测试这个配置的真实平均实力与稳健性。

🏎️ 行业共识比喻：

  • Configuration (配置) 就像赛车的发动机排量、轮胎抓地力等级（这是我们可以主动调整和优化的设计参数）。
  • Seed (种子) 就像比赛当天的天气状况、风向、赛道上的小石子分布（这是不可控的随机噪声）。
  • 实验逻辑：我们要寻找一套最好的汽车配置（Configuration）。为了测试这套配置是不是真的好，我们必须让它在下雨天、刮风天、大晴天等 30 种不同的随机天气（Seed 1 到 30）下都跑一遍，最后再用统计假设检验算一个总账！

第一步：为什么均值和标准差不够科学？

在处理上述 30 次随机跑（不同 Seed）得出的数据时，均值 (Mean/Average) 和 标准差 (Standard Deviation) 具有致命缺陷：

  1. 异常值敏感 (Sensitive to Outliers): 讲师强调，极端的运行结果会严重扭曲均值。
  2. 无法反映分布形状: 均值相同并不代表两个算法性能一致。
  3. 误差棒重叠: 当两组数据的误差棒（Error Bars）严重重叠时，肉眼无法判定显著性。

💡 解决方案：使用中位数和四分位数 (Medians & Quartiles)

  • 中位数 (Median): 排序后的中间值，对异常值具有鲁棒性 (Robust)。
  • 第 1/3 四分位数 (1st/3rd Quartiles): 描述数据 25% 和 75% 的位置，能更真实地反映波动程度。

第二步：构建统计假设 (Formulating Hypotheses)

在确保评估开销和环境绝对公平 (Fair Comparison) 的前提下，建立统计检验游戏规则：

  • 原假设 $H_0$ (Null Hypothesis): 两组算法/配置性能没有显著差异。
  • 备择假设 $H_1$ (Alternative Hypothesis): 两组算法/配置性能有显著差异。

第三步：选择正确的统计检验方法 (Test Selection)

选择方法取决于数据分布和实验设计，可以参考下表：

数据类型 独立样本 (Unpaired) 配简样本 (Paired) 场景
参数检验 (Parametric) Unpaired t-test Paired t-test 符合正态分布
非参数检验 (Non-parametric) Mann-Whitney U / Wilcoxon rank-sum Wilcoxon signed-rank 不符合正态分布
多组对比 (N-groups) Kruskal-Wallis Friedman N > 2 组不同配置对比
  • 参数检验 vs. 非参数检验: 参数检验（如 t-test）要求数据符合正态分布（Normality Assumption）。讲师指出：“Non-parametric tests are more popular because in reality majority of the cases do not follow the normal distributions.”
  • 配对 vs. 非配对: 如果两个不同管线配置在每次运行时均采用完全一样的环境切分、同一组初始随机种群或相同的数据折叠（Identical Initial Conditions），则应使用配对检验 (Paired Test)，其统计检定力（Power）更强。
    这张表格（课件第 41-42 页）是整个统计学检验模块的“交通指挥官” 。当你在智能软件工程（ISE）中跑完了好几种 Configurations 并在不同的 Seeds 下拿到了实验数据（Observations）后，应该选哪个统计检验方法，完全由这张表决定 。

为了让你彻底吃透它，我们不机械地背表，而是从三个决策维度（决策树的三个分叉点）来详细拆解它的选择取舍（Trade-offs）。

🧭 维度一：组数（Columns的分流）—— 2 groups vs n groups (n>2)

首先看表格的列，这取决于你的实验设计里同时拿出来对比的 Configuration（配置管线）有几个 。

  1. 2 groups（两组对比）

  • 场景：你只有两个选手在做 A/B 测试 。例如：Config A (开启降维) vs Config B (不开启降维)。

  • 可供选择的检验：t-test 家族或 Wilcoxon 家族 。

  1. n groups (n>2)（多组对比）

  • 场景：你在进行多方案选型，有 3 个或以上的选手同时打架 。例如：Config A (线性模型) vs Config B (树模型) vs Config C (深度学习) 。

  • 可供选择的检验：ANOVA、Kruskal-Wallis、Friedman 。

  • 🚨 核心取舍（Critical Trade-off）： 如果手里有 4 组数据，千万不要两两之间做 6 次两两检验（Pairwise Comparisons） ！因为课件强调了，连续做多次检验会导致 第一类错误（Type 1 Error，即假阳性）的概率暴增（从 5% 飙升到极高） 。因此，面对多组配置，必须先用右边这一列的全局检验（Global Test）一网打尽 。

📈 维度二：数据分布（Rows的一级分流）—— Parametric vs Non-parametric

这是由你拿到的性能指标数据的分布特征决定的，也是统计学里最核心的硬性约束 。

  1. Parametric（参数检验）

  • 前提条件：数据必须符合正态分布（Normality Assumption） 。也就是说，30次运行出来的覆盖率或者延迟，画成直方图必须是一个两边对称、中间高的钟形曲线 。

  • 入选方法：Unpaired t-test 、Paired t-test 、ANOVA 。

  • 👍 优势：检定力更强（More powerful） 。如果数据确实符合正态分布，它对配置之间“微小差异”的捕捉能力极其敏锐，能极好地探测出显著差异 。

  • 👎 劣势：极其脆弱（Highly affected by violations） 。如果你的随机算法跑出来的数据不符合正态分布（比如 ISE 中常出现的严重偏斜、双峰数据），强行使用参数检验会导致算出来的 P值 完全是错的 。

  1. Non-parametric（非参数检验）

  • 前提条件：不需要任何分布假设（No normality） 。数据长成什么奇形怪状都可以用（比如有严重的异常值、极度偏斜） 。

  • 入选方法：Wilcoxon rank-sum 、Wilcoxon signed-rank 、Kruskal-Wallis 、Friedman 。

  • 👍 优势：极度安全、受欢迎（More popular） 。因为智能软件工程中的 stochastic algorithms（随机算法）在不同种子下跑出来的性能结果，绝大多数都不能保证是完美的正态分布 。非参数检验通过将真实数据转化为排名（Ranks）进行计算，天然免疫异常值的干扰 。

🔗 维度三：血缘关系（Rows的二级分流）—— Unpaired vs Paired

这取决于你在写代码进行实验时，是怎么把随机种子（Seeds）和配置（Configurations）组合运行的 。

  1. Unpaired / Independent（独立样本检验）

  • 实验设计：各个配置的运行“各跑各的，互不干扰” 。 例如：Config A 用了种子 1-30 跑了 30 次；Config B 用了全新的种子 31-60 跑了 30 次 。它们之间在同一跑上没有任何环境共性。

  • 选用方法：Unpaired t-test 、Wilcoxon rank-sum / Mann-Whitney U test 、Kruskal-Wallis 。

  1. Paired / Related（配对/相关样本检验）

  • 实验设计：控制变量法，各个配置在运行期间强行锁定相同的外部随机性（环境共享） 。 例如：第 1 轮实验，把 Seed 42 同时喂给 Config A 和 Config B（让它们在完全相同的初始种群和数据切分下对抗）；第 2 轮实验，再把 Seed 99 同时喂给它们……

  • 选用方法：Paired t-test 、Wilcoxon signed-rank 、Friedman 。

  • 🚨 核心取舍（Critical Trade-off）： 课件第 51 页一针见血地指出：“Paired tests use more information, being more powerful” 。因为配对设计锁定了外部环境的随机性（把考场噪音抵消了），它利用了“同一轮中环境相同”这一极高价值的信息，所以其发现配置差异的能力显著高于独立检验 。因此，在写实验代码时，能做成 Paired 设计，就千万不要做成 Unpaired ！

🛠 总结：在 Intelligent SE 中最常用的四大黄金组合

由于智能软件工程的随机算法结果通常不符合正态分布，我们在实战中绝大多数都在使用 Non-parametric（非参数） 这一半张表 ：

  1. 两套配置对比，各跑各的种子 $\rightarrow$ Wilcoxon rank-sum test (Mann-Whitney U) 。

  2. 两套配置对比，共享同一批种子 $\rightarrow$ Wilcoxon signed-rank test（检定力强，最常用） 。

  3. 多套配置对比，各跑各的种子 $\rightarrow$ Kruskal-Wallis test，跑完再接 Dunn 事后检验 。

  4. 多套配置对比，共享同一批种子 $\rightarrow$ Friedman test，跑完再接 Nemenyi 事后检验 。

现在，对着这张图重新审视你的实验，是不是每一个格子对应的统计学取舍都清清楚楚了？接下来，你想去深入推导某一个具体检验（比如两组配对的 Wilcoxon）的输出，还是去探索大纲第一部分的多目标质量指标？

第四步：解读 P 值与多组比较校正

  • P 值 (P-value): 如果 $P \le 0.05$（显著性水平 $\alpha$），则拒绝原假设 $H_0$，认为差异具有统计显著性。
  • 第一类错误 (Type 1 Error): 俗称“假阳性”，即本无差异却误判为有差异。
  • 多组比较问题: 如果我们有数十种 Pipeline 组合（如 Config A, B, C, D…），并不断进行两两对比（Pairwise comparisons），犯第一类错误的概率会激增。
    • 解决方案 A (校正): 使用 Bonferroni 校正（显著性阈值除以对比次数）或更灵活的 Holm-Bonferroni 校正。
    • 解决方案 B (推荐：全局检验): 如 2.1 节所述，先运行全局检验（如 Kruskal-Wallis），若 P 值显著，再进行事后检验 (Post-hoc Test)（如 Dunn 检验或 Nemenyi 检验）来精准锁定最终的胜出配置。

3. 实验总结与报告准则

在撰写研究报告或论文时，应遵循以下逻辑：

  1. 展示描述性统计 (Descriptive Statistics): 使用箱线图 (Boxplots) 展示各种配置跑完 30 轮（30 种 Seed）后的中位数和四分位数。
  2. 报告检验结果: 指出使用的检验方法（如 Kruskal-Wallis 结合 Dunn 事后检验）及其 P 值。
  3. 综合结论: 只有当 P 值显著时，才结合指标数值（如“配置 A 的中位数显著高于配置 B”）得出“深度学习管线 A 显著优于传统管线 B”的结论。

思考题解析：
若要对比 3 种配置（非正态分布），且每个配置是独立运行（Unpaired）：

  1. 首先运行 Kruskal-Wallis 检验 确定是否存在全局差异。
  2. 若 P 显著，运行 Dunn 检验 进行事后分析，确定哪两组配置之间有显著区别。
