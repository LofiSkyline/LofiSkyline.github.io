第 10 讲：元数据安全与隐私保护搜索 (Metadata Security & Privacy-Preserving Search)

第 10 讲：元数据安全与隐私保护搜索 (Metadata Security & Privacy-Preserving Search)

在之前的课程中，我们主要关注内容安全 (Content Security)（即加密数据本身）。然而，仅仅加密数据是不够的，因为元数据 (Metadata)（如所有者信息、时间戳、访问权限、密文长度）和访问模式 (Access Patterns)（即哪些具体的块被检索）可能会向好奇的云服务商或攻击者泄露大量信息。

1. 可搜索加密 (Searchable Encryption, SE)

可搜索加密 (SE) 允许客户端将加密数据存储在服务器上，并在服务器不了解文档内容或搜索关键字的情况下执行关键字搜索。

1.1 索引设计 (Index Design)

为了在服务器端实现高效搜索，我们不会直接搜索加密文档。相反，我们使用可搜索索引 (Searchable Index)。主要有两种结构：

  1. 正向索引 (Forward Index): 将文档 ID (Document ID) 映射到关键字列表。搜索时需要扫描所有文档条目，效率较低（复杂度为 $O(\text{Documents} \times \text{Keywords})$）。
  2. 反向索引 (Inverted Index): 将关键字 (Keyword) 映射到文档 ID 列表。这种方式速度快得多（复杂度为 $O(1)$ 或 $O(\text{Keywords})$），是大多数 SE 方案的首选。

对称密钥 SE 机制 (Symmetric Key SE Mechanism):
在对称密钥 SE 方案中，客户端在本地构建索引。对于反向索引，每个关键字 $W$ 都会转换为一个“陷门” (Trapdoor, 即搜索令牌)。与 $W$ 关联的文档 ID 使用从关键字本身派生的密钥进行加密。

“服务器拥有这一切… 服务器只看到某个关键字的密文。然后服务器继续尝试解密所有内容… 凡是解密成功的（因为密钥匹配），就会返回；解密失败的则不返回。” (转录文本 wk10-1)

1.2 填充过程 (Padding Process)

填充对于隐藏搜索结果大小 (Search Result Size) 至关重要。如果关键字 “CIA” 返回 2 个文档，而 “NIXON” 返回 50 个，服务器就能了解哪些关键字更常见，从而可能导致统计分析攻击。

  • 过程 (Process): 客户端为索引中的每个关键字条目定义一个最大文档 ID 数量。如果某个关键字出现的文档较少，客户端会向列表中添加虚假条目 (Dummy Entries)（加密的随机噪声）。
  • 结果 (Result): 无论搜索哪个关键字，返回给客户端的加密块数量都是相同的，从而掩盖了关键字的真实出现频率。

1.3 局限性与风险 (Limitations and Risks)

尽管 SE 效率很高，但它在安全权衡上存在显著缺陷：

  1. 确定性陷门 (Deterministic Trapdoors): 大多数基础 SE 方案是确定性的。两次搜索 “Top-Secret” 会产生相同的陷门。服务器可以据此了解搜索模式 (Search Pattern)（即相同术语何时被搜索、搜索了多少次）。
  2. 访问模式泄露 (Access Pattern Leakage): 这是最关键的风险。SE 虽然保护了索引搜索过程，但一旦服务器返回了文档 ID，客户端通常会去下载实际的加密文档。
    • “免责声明”风险: “即使搜索查询的响应是以隐私保护的方式完成的，服务器仍然可以通过简单地观察客户端接下来的操作（例如监控客户端将要访问/下载的电子邮件）来了解查询结果。” (课件第 57 页)。

2. 不可知内存访问 (Oblivious RAM, ORAM)

ORAM 是一种更强大（也更昂贵）的原语，旨在彻底隐藏访问模式 (Access Patterns)。它确保服务器无法分辨客户端是在读取还是写入，无法得知哪个块被访问，甚至无法得知是否在重复访问同一个块。

2.1 树形结构存储 (Tree-Based Structure)

现代 ORAM（如 PathORAM）将服务器端存储组织为一棵由桶 (Buckets) 组成的二叉树。

  • 桶 (Buckets): 树中的每个节点都是一个“桶”，包含固定数量 ($Z$) 的数据块。
  • 逻辑与物理 (Logical vs. Physical): 虽然逻辑上是一棵树，但在服务器磁盘上它是以扁平数据结构 (Flat Data Structure, 如数组) 存储的，以防通过物理内存地址泄露树的层次结构。
  • 路径映射 (Path Mapping): 每个数据块被随机映射到树的一个特定叶节点 (Leaf Node)。该块可以存在于从根节点到该叶节点路径上的任何位置。

2.3 运行机制与 PathORAM 算法 (Mechanism)

读取或写入块 (ID 为 $a$) 的过程如下：

  1. 查找 (Lookup): 客户端查询本地的位置表 (Position Map)，找到块 $a$ 当前映射到的叶节点 $L$。
  2. 重映射 (Remap): 客户端立即为块 $a$ 分配一个新的随机叶节点 $L’$。
  3. 读取路径 (Read Path): 客户端下载从根节点到叶节点 $L$ 的整条路径，存入本地的 Stash（一个小型本地缓存结构）。
  4. 处理 (Process): 客户端在 Stash 中解密块，找到块 $a$，如果是写入操作则更新其内容。
  5. 回写 (Write Back): 客户端重新加密所有块，并将它们写回同一条路径（根节点到 $L$）。关键点在于，它会将 Stash 中的块尽可能深地推向它们被分配的新叶节点。
sequenceDiagram
    participant Client as 客户端
    participant Map as 位置表 (Position Map)
    participant Server as ORAM 树 (服务器)
    
    Client->>Map: 查询块 ID (a)
    Map-->>Client: 当前路径: 路径 L
    Note over Client: 重映射: ID(a) -> 新路径 L'
    Client->>Server: 请求路径数据 (根节点 -> L)
    Server-->>Client: 返回整条路径 (Z * 层数 个块)

    Note over Client: 更新或读取块 (a)
    Note over Client: 使用 Stash 中的块重新加密
    Client->>Server: 回写路径数据 (根节点 -> L)

“每一次读或写操作，实际上都包含了读和写两个动作… 这种访问操作具有巨大的开销，目的是为了掩盖到底哪个数据被访问了。” (转录文本 wk10-1)。

2.4 短板与性能开销 (Shortcomings & Performance Overhead)

  • 客户端存储: 客户端必须维护位置表 (Position Map)（可能很大，但可以通过递归 ORAM 处理）和 Stash。
  • 单用户限制: PathORAM 传统上仅限单用户；通过代理实现多用户访问会引入定时攻击风险。

例题：PathORAM 带宽开销计算 (Slide 77-80)

1. 题目已知条件 (The Setup)
假设我们在云端有一个 128GB 的数据库，使用的 PathORAM 参数如下：

  • $S$ (Block size / 数据块大小): 64KB
  • $Z$ (Blocks per bucket / 每个桶的容量): 5 个 blocks
  • $L$ (Levels / 二叉树的层数): 20 层
  • 用户目标: 想要从云端读取一个名为 SecretDocument.txt 的文档，它的大小是 1MB。

2. 逐步计算过程 (Step-by-Step Calculation)

  • 第一步：计算文档被切成了多少个 Block ($N$)
    在 ORAM 中，数据不能作为一个整体传输，必须切成固定大小的块。
    已知 $1MB = 1024KB$。
    文档包含的块数 $N = 1024KB / 64KB = 16$ 个 Blocks。
  • 第二步：理解“读取一个 Block”的真实代价
    在 PathORAM 的机制里，你不能只下载你想要的那个 Block。为了掩人耳目，每次访问 1 个 Block，你必须把整条路径 (Path) 上的所有内容全部下载下来（并且稍后还要全部写回去）！
    • 一条路径有多少层？ $L = 20$ 层。
    • 每层（每个 Bucket）有几个块？ $Z = 5$ 个。
    • 每个块多大？ $S = 64KB$。
    • 下载一条路径的数据量 = $L \times Z \times S = 20 \times 5 \times 64KB = 6400KB$ (即 6.25MB)。
  • 第三步：计算总带宽消耗 (Total Bandwidth)
    你总共需要读取目标文档的 $N=16$ 个 Blocks。每次读取都要下载/上传一条完整的路径。
    总带宽 = $N \times S \times Z \times L$
    代入数字：$16 \times 64KB \times 5 \times 20 = 102,400KB$
    换算成 MB：$102,400KB / 1024 = 100MB$。

3. 核心结论 (The Takeaway)
为了下载一个仅仅 1MB 的真实文档，在 PathORAM 的机制下，你的网络实际传输了 100MB 的数据！这相当于产生了 100 倍的带宽开销 (100x Overhead)。这个极端的例子完美解释了为什么 ORAM 至今难以在普通商业场景中大规模普及。

3. 对比展示：ORAM vs. 可搜索加密 (SE)

特性 (Feature) 可搜索加密 (SE) 不可知内存访问 (ORAM)
主要目标 (Goal) 在预计算索引中进行安全关键字搜索。 隐藏所有访问模式（读/写/频率）。
安全级别 (Security) 保护搜索词和结果；但会泄露访问模式。 全面保护访问模式和数据内容。
效率 (Efficiency) 高（毫秒级查询）。 低（巨大的带宽和延迟开销）。
典型用途 (Use Case) 邮件/数据库搜索 (如 OXT, 2Lev 方案)。 高安全需求的医疗/基因数据，隐私信息检索 (PIR)。
元数据保护 保护关键字相关的元数据。 保护连续访问之间的关联性。

总结 (Summary):
可搜索加密 (SE) 就像是一个带锁的索引。你给它一个加密的令牌，它还你匹配的结果。它非常快，但存在“后续行为泄露”风险：服务器虽然不知道你搜了什么，但看你下载了哪个加密包，就能猜出搜索结果。

不可知内存访问 (ORAM) 则更像是一个不断洗牌的魔术。每次你想看一个球，魔术师都会把整排杯子（整条路径）都拿走，在后台把球的位置换掉，然后再把整排杯子放回去。服务器完全不知道你在看哪个，甚至不知道你在看还是在换，但代价是极高的带宽消耗（通常是原文件大小的百倍）。
