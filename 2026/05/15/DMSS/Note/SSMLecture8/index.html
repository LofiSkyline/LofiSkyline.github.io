Local Data Security: Full Disk Encryption (FDE)

Local Data Security: Full Disk Encryption (FDE)

1. Introduction to Data Security Types

数据安全主要分为 Data at rest（静态数据，存储在硬盘、数据库或备份中）和 Data in use（在使用中的数据，正在被 CPU 或 RAM 处理）。全盘加密（Full Disk Encryption, FDE）的目标是在设备关闭或被盗时保护静态数据，使其对未授权者不可访问。

2. The Bootstrap Dilemma (启动的困境)

A core challenge in FDE is the “CPU’s Dilemma”: If the entire disk is encrypted, the CPU cannot read the decryption algorithm because it is also locked inside the encrypted partition.

  • The Problem: “Where is my decryption algorithm?” (我的解密算法在哪？)
  • The Solution: The disk is partitioned. A small part called the Boot/Kernel Partition (or System Drive) remains unencrypted or resides in a pre-boot environment. This area contains the Bootloader and Kernel, which “know” the AES algorithm and can interact with hardware to fetch the keys.

3. Key Hierarchy and Storage (密钥体系与存储)

A. Core Keys (核心密钥)

  • DEK (Data Encryption Key / Drive Key):
    • Role: The bottom-layer symmetric key that actually encrypts the 4K blocks of data on the drive.
    • Location: Stored on the drive in an encrypted format at a random location. It “never leaves the device” in its raw form except when loaded into the volatile memory (RAM/Kernel Space) during operation.
    • Persistence: It is Static (静态的). Once generated, it remains the same unless the drive is wiped. 除非驱动器被恢复出厂设置，加密格式的 DEK 永远存在磁盘上。
  • AK (Authentication Key) / VMK (Volume Master Key):
    • Role: Used to “wrap” (encrypt) the DEK. Only by obtaining the AK can the system decrypt the DEK and subsequently the drive.
    • Verification: A Hash of AK is stored on the drive to verify if the user-provided credentials are correct.

B. Storage Strategies (密钥存放权衡)

  1. USB Stick: Simple but vulnerable to physical theft.
  2. TPM (Trusted Platform Module):
    • Sealing: The TPM “seals” (封印) the AK/VMK inside its hardware memory.
    • Integrity Check: The TPM only releases the key if the PCR (Platform Configuration Registers) show that the early boot files (Bootloader/Kernel) are Unmodified. This creates a Root of Trust.

4. Key Derivation Functions (KDF) - 密钥派生

Humans are bad at remembering high-entropy keys. KDFs bridge the gap between a low-entropy password and a high-quality cryptographic key。

  • Process: $Key \leftarrow Hash(Password, Salt) \times Iterations$
  • Salt (盐值):
    • Adds randomness to the input.
    • Purpose: Prevents pre-computation attacks (Rainbow Tables) and ensures that two users with the same password have different derived keys.
  • Iterations (Loops):
    • The hash is run thousands of times to introduce an artificial delay (引入人为延迟)。
    • Purpose: Slows down brute-force attacks by making each password guess computationally expensive (e.g., 1 second instead of 1 microsecond).

5. From Factory Setup to User Encryption (出厂设置到用户激活)

Phase 1: Manufacturer Signing (厂商签名阶段)

厂商在出厂前，使用其 Private Key 对 Bootloader 和 Kernel 进行签名。对应的 Public Key 预埋在硬件（TPM/TEE）中。

Phase 2: Initial Boot & Baseline (初次启动)

  1. Integrity Verification: TPM 使用 Public Key 验证启动代码。如果一致（Unmodified），则建立信任。
  2. DEK Generation: 系统随机生成一个 128/256-bit 的 DEK (Data Encryption Key)。此时 DEK 处于“裸奔”状态，或由默认密钥保护。

Phase 3: User Setup (用户加密阶段)

  1. User Input: 用户设置密码。
  2. KDF Application: 系统通过 KDF 和随机生成的 Salt 计算出用户的 AK (Authentication Key)。
  3. Wrapping: 系统在 RAM 中用 AK 加密 DEK，生成 enc_drive_key。
  4. Persistent Storage: 将 enc_drive_key、Salt 和 Hash of AK 写入磁盘。此时，DEK 被正式“锁入保险箱”。

6. Password Update: Behind the Scenes (更换密码的具体步骤)

Changing your password does not require re-encrypting the whole drive. The system performs a “key re-wrapping” process:

  1. Unlock: User enters the old password. The system derives the Old AK, decrypts the enc_drive_key in RAM to get the Raw DEK.
  2. New Key Derivation: User enters the new password. The system generates a New Salt and derives a New AK via KDF.
  3. Re-wrap: The system uses the New AK to encrypt the same Raw DEK.
    • $enc_drive_key_{(new)} = Encrypt(New_AK, DEK)$
  4. Overwrite: The new enc_drive_key, new Salt, and new Hash of AK are written to the disk, overwriting the old ones.
  • Conclusion: The DEK is static. Only the “outer layer” (AK) that protects it changes.

7. FDE Disk Structure & Layout

A. General FDE Structure (LUKS/dm-crypt style)

  • MBR (Master Boot Record): Sector 0. Contains partition tables and basic boot code.
  • HEADER (Cleartext): The “Instruction manual” for decryption. It includes:
    • Encryption Type (e.g., LUKS)
    • Algorithm (e.g., AES-XTS)
    • IV (Initialization Vector)
    • Hardware requirements (e.g., TPM)
  • Encrypted Logical Volume: The actual data blocks (4K blocks).

B. BitLocker Specific Structure

BitLocker uses a dual-drive split for a more robust “Transparent” experience:

  1. System Drive (Unencrypted):
    • Contains: Bootloader, Kernel, and the Decryption Algorithms.
    • Role: Bootstraps the system and performs the “Measurement” of hardware.
  2. OS Drive (Encrypted):
    • Contains: System files, User data, and Cache/SWAP.
    • Role: Stays encrypted at rest.
  • Key Storage: The key is “Sealed” in the TPM. If the System Drive is tampered with (e.g., bootloader replaced), the TPM detects the change in PCRs and refuses to release the key.

File-based Encryption (FBE)

基于文件的加密（FBE）允许对不同的文件或目录使用不同的密钥，提供细粒度（Granularity）控制。

1. iOS Protection Classes (Apple 方案)

iOS 的 FBE 建立在 SEP (Secure Enclave Processor) 硬件隔离之上，定义了四种保护类（Protection Classes）。

A. 详细分类介绍

类 (Class) 含义与生成方式 用途与存放位置
Class D 硬件根密钥 (No Protection)。由 SEP 持有的唯一设备密钥 ($UID$)。 存放于 SEP。作为所有类密钥的底层支柱，参与所有类密钥的派生，确保数据与物理硬件绑定。
Class C 默认保护 (Default Protection)。在第一次解锁（First Unlock）后派生。 存放于 RAM（解锁后）。用于保护系统文件和基础 App 数据。在设备解锁一次后，即便再次锁屏，该密钥也驻留在 RAM 中。
Class A 完全保护 (Complete Protection)。由 $Passcode + Class D$ 通过 KDF 派生。 锁屏 10 秒后从 RAM 抹除。用于保护核心隐私。每次访问都必须确保设备处于解锁状态。
Class B 非对称保护 (Protected Unless Open)。包含一对公私钥。 公钥存 RAM，私钥锁屏即消失。允许在锁屏状态下接收/写入新数据（用公钥加密），但只有解锁后才能读取（释放私钥）。

B. iOS 工作流程案例：访问一张 Class A 的照片

  1. 解锁手机: 用户输入密码，SEP 结合内部 $UID$ (Class D) 现场计算出 Class A Key 并送入 RAM。
  2. 解析元数据: 系统使用已在 RAM 中的 Class C Key 解开文件系统的 Header，读取照片的 Metadata。
  3. 获取封装密钥: 在 Metadata 中找到该照片独有的 Wrapped File Key（被 Class A 包裹的 File Key）。
  4. 拆封过程: 系统调用 RAM 中的 Class A Key 对 Wrapped File Key 进行解密，还原出明文的 AES File Key。
  5. 读取内容: 内核使用该 AES File Key 实时解密磁盘上的照片数据块。

2. Android FBE Structure (Android 7.0+ 方案)

Android 采用双分区架构，依赖 TEE (Trusted Execution Environment) 进行密钥管理。

A. CE 与 DE 分区

  • DE (Device Encrypted) Storage: 开机即用。存储闹钟、电话等，确保手机重启后即便不输入密码也能提供基础功能。
  • CE (Credential Encrypted) Storage: 隐私存储。只有用户输入密码并成功验证后，对应的密钥才会被释放。

B. Android 密钥体系 (Key Hierarchy)

密钥名称 生成方式 用途 存放位置
Hardware Root Key 厂商预埋在硬件中。 信任链的最底层。 TEE (不可导出)
Stretched Credential $Password + Salt \xrightarrow{KDF} …$ 作为中间凭据。 RAM (临时存在)
KEK (Key Encryption Key) 在 TEE 内结合 Stretched Credential 和 Root Key 算出。 拆快递的“剪刀”。专门用来解开磁盘上的 Master Key 包裹。 TEE (运算生成)
Wrapped Master Key 安装/设置密码时生成的加密 Blob。 Master Key 的持久化加密形式。 Disk (磁盘持久化)
Master Key (CE/DE) 系统用 KEK 在 TEE 内解开 Wrapped Master Key 得到。 领域总管。负责派生所有具体文件的 AES 密钥。 Kernel Keyring (RAM)
File Key (AES Key) $KDF(Master Key, Inode Info)$ 苦力。真正负责 4K 数据块加解密的对称密钥。 Kernel RAM (动态生成)

C. Android 工作流程案例：访问一个 CE 分区的文件

  1. 用户认证: 用户输入 PIN/密码。系统通过 KDF 将密码拉伸为 Stretched Credential。
  2. 生成 KEK: Stretched Credential 被送入 TEE，结合硬件不可导出的 Root Key，计算出该用户本次会话的 KEK。
  3. 解密 Master Key: 系统从磁盘加载 Wrapped Master Key (Blob)，送入 TEE。TEE 使用 KEK 对其进行解密，得到明文的 CE Master Key。
  4. 注入内核: 明文 Master Key 被安全地从 TEE 传递到 Kernel Keyring（内核密钥环）中。
  5. 派生 File Key: 当用户尝试打开一个特定的隐私文件时，内核提取该文件的 Inode（索引节点）信息，结合 Master Key 通过算法派生出该文件专属的 AES File Key。
  6. 实时加解密: 内核的 Crypto API 使用该 File Key 对磁盘上的数据块进行解密，并将明文数据送入应用层。

3. iOS vs. Android FBE 对比

维度 iOS (SEP 模型) Android (dm-crypt/Keyring 模型)
硬件集成 强集成。SEP 位于数据流水线上，密钥物理隔离在 SEP 内，甚至不经过 CPU/内核。 弱集成。依赖 TEE 管理 KEK，但具体加解密操作通常在 CPU/内核空间由软件完成。
优雅性 (Class B) 极高。非对称加密巧妙解决了锁屏下载/后台写入的问题。 相对复杂。通过多 Master Key 结构实现，但没有原生对应 Class B 的优雅逻辑。
安全性 极强。硬件级隔离确保即便内核被 Root，攻击者也难以提取处于 Class A 状态的密钥。 强。但由于加解密在内核空间，Root 后的内核可能面临内存注入或 API 挂钩的风险。
灵活性 固定四种策略，系统级统一强制执行。 非常灵活。支持无限分层，允许厂商根据硬件定制 TEE 模块，通用性极佳。

Summary for Revision:

  • FBE 核心痛点: 元数据（Metadata）泄露（文件名、大小等可见）。
  • iOS 核心优势: SEP 硬件流水线隔离与 Class B 非对称写入。
  • Android 核心优势: CE/DE 分区隔离与灵活的 Keyring 管理。
  • 共同点: 都是通过 Key Wrapping (密钥包裹) 和 KDF 动态派生 避免一破全破。
