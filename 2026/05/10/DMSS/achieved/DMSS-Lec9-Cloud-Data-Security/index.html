云数据安全 (Cloud Data Security)

云数据安全 (Cloud Data Security)

云数据安全探讨的是如何在不完全信任云服务提供商（CSP）的情况下，保护云端数据的机密性、完整性和可用性。

1. 云计算架构与动机 (Cloud Computing Fundamentals)

云按照部署模式主要分为四种类型：

  • 公共云 (Public cloud)：资源由公众共享。
  • 私有云 (Private cloud)：组织内部专属拥有和维护。
  • 社区云 (Community cloud)：具有共同诉求的群体共同管理。
  • 混合云 (Hybrid cloud)：结合公共云与私有云，例如利用私有云处理敏感数据，用公共云进行海量存储。

企业采用云计算的主要动机包括：简单性 (Simplicity)、成本效益 (Cost)、安全性 (Security)、系统弹性 (Resilience)、灵活性 (Flexibility) 以及创新速度 (Pace of innovation)。

2. 云环境下的核心安全威胁 (Main Cloud Security Issues)

根据云安全联盟（Cloud Security Alliance, CSA）的定义，云环境面临 12 大主要安全挑战：

  1. 数据泄露 (Data breaches)：未经授权访问受保护的敏感信息（例如 Facebook 或 Equifax 泄露事件）。
  2. 弱身份、凭证和访问管理 (Insufficient identity, credential and access management)：例如由于未采用多因素认证 (MFA) 或把密钥硬编码在 GitHub 上导致的账户被盗。
  3. 不安全的接口与 API (Insecure APIs)：云服务的配置和管理高度依赖 API，API 的漏洞会直接危害整个云服务（例如 IRS 数据泄露）。
  4. 系统和应用漏洞 (System & application vulnerabilities)：攻击者利用操作系统、内核或库中的漏洞入侵系统（例如 Bash 的 Shellshock 和 OpenSSL 的 Heartbleed 漏洞）。
  5. 账户劫持 (Account hijacking)：通过网络钓鱼或跨站脚本 (XSS) 等手段窃取凭据。
  6. 恶意内部人员 (Malicious insiders)：现任或前任员工滥用访问权限，损害数据机密性。
  7. 高级持续性威胁 (Advanced Persistent Threats, APTs)：攻击者长期潜伏在系统中，隐蔽地实现商业或政治目的。
  8. 数据丢失 (Data loss)：无论是恶意攻击、意外删除，还是由于灾难导致的数据永久丢失。
  9. 尽职调查不足 (Insufficient due diligence)：企业在迁移到云或选择 CSP 之前没有进行充分的风险评估。
  10. 滥用云服务 (Abuse and nefarious use of cloud services)：攻击者利用云的强大计算资源发动 DDoS 攻击、挖掘加密货币或托管恶意内容。
  11. 拒绝服务 (Denial of service, DoS)：通过耗尽系统资源（如 CPU、内存、带宽）阻止合法用户访问服务。
  12. 共享技术漏洞 (Shared technology issues)：由于 IaaS, PaaS, SaaS 是多租户架构，底层硬件（CPU缓存等）和虚拟机管理程序 (Hypervisor) 的隔离如果不彻底，可能引发跨虚拟机攻击（如 VENOM 漏洞）。

3. 攻击者模型 (Attacker Models)

在设计云安全协议时，通常会假设不同能力的攻击者模型：

  • 诚实模型 (Honest)：并非真正的攻击者，完全遵守协议。
  • 诚实但好奇 (Honest-but-curious)：云环境中最常用的模型。攻击者遵守协议的执行流程，但是一个被动攻击者 (passive attacker)，会尽可能地窥探和收集数据。
  • 恶意但谨慎 (Malicious-but-cautious)：比诚实但好奇更强大。他们是主动且隐蔽的 (active and covert)。他们可能会篡改数据，但前提是试图保持不被发现（不留下可验证的证据）。这通常是用来建模云服务提供商 (CSP) 的最佳模型——他们可能会为了商业利益偷窥或利用数据，但又必须小心谨慎以保住客户。
  • 恶意模型 (Malicious)：最强大的攻击者（例如 Dolev-Yao 模型），对网络和资源拥有完全控制权，会不择手段地访问、篡改或破坏数据。

4. 针对云提供商的保密性 (Confidentiality from the Cloud Provider)

现代云安全研究的“圣杯” (Holy grail) 是：如何在享受云计算带来的便利的同时，不向云提供商泄露明文数据。如果我们不需要信任云服务商的员工或子承包商，那么即使云端被黑客攻破，我们的数据依然是安全的。

为了实现这一目标，学术界和工业界提出了多种前沿架构，课件中重点介绍了 CryptDB 和 ConfiChair 两个系统。

4.1 加密数据库系统：CryptDB

CryptDB 是一个位于应用程序和底层数据库（如 MySQL）之间的安全中间件。

  • 工作原理：它将所有数据加密后存入数据库。当应用程序发出 SQL 查询时，CryptDB 代理服务器 (Proxy) 会将其转换为加密条件下的 SQL 查询。数据库在密文状态下执行查询并返回密文结果，最后由代理服务器解密后返回给用户。
  • 洋葱加密 (Onion Encryption)：为了在加密数据上进行数据库操作，CryptDB 为每个数据列应用了多层加密（像洋葱一样）。随着所需操作的不同，代理服务器会剥开外层加密，暴露所需的内层功能：
    • RND (随机加密)：最安全，提供最高的机密性，但不支持任何数据库操作。
    • DET (确定性加密)：允许进行等值比较 (equality selection) 和 等值连接 (equality join)。
    • OPE (保序加密, Order-preserving encryption)：密文的顺序与明文顺序相同 ($x < y \implies enc(x) < enc(y)$)，允许进行排序 (sort) 和 范围连接 (range join)。
  • 缺点/权衡：由于系统必须满足查询需求，随着时间的推移，数据会默认退化到允许该操作的最弱安全级别。因此它不完全符合“故障导向安全 (Fail to Safe)”的原则。
  • 密钥链 (Key Chaining)：密钥来源于用户的密码。当用户登录时，代理获取密钥并代表用户进行操作；注销后，密钥被销毁。即使代理服务器被短暂攻破（Threat 2），攻击者也无法获取未登录用户的数据。

4.2 安全的会议管理系统：ConfiChair

类似于 EasyChair 的 SaaS 会议管理系统，但解决了 CSP 可以直接访问所有论文和评审记录的问题（这些记录可用于形成对学者的评价画像）。

  • 加密机制：利用 HTML5 和 JavaScript 在用户的浏览器中（In-browser）完成所有的加密操作。云端仅存储加密数据（No un-encrypted cryptographic material）。
  • 密钥钱包 (Keypurse)：每个用户（主席、作者、审稿人）都有一个由密码保护的 Keypurse，用于存储非对称密钥和会议对称密钥。
  • 系统安全属性：
    1. 机密性 (Secrecy)：ConfiChair（云端）不知道论文内容、审稿意见、打分和讨论。
    2. 不可链接性 (Unlinkability)：云端无法知道特定审稿人 R 审阅了特定作者 A 的论文。(注：课件中提到，实际上实现严格的不可链接性非常困难，目前采用将论文分组打包下载的方法并不是非常安全)。
