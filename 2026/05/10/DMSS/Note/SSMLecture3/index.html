Lecture 3: Asset Registers & Cloud Assets (资产登记册与云资产)

Lecture 3: Asset Registers & Cloud Assets (资产登记册与云资产)

1. Recap: Scope and Definitions (回顾：范围与定义)

The Information Security Management System (ISMS) depends on a clear structure (信息安全管理体系依赖于清晰的架构):

  • Policies (策略): Define objectives and outcomes (定义目标和预期结果).
  • Procedures (程序): Define methods and measures (定义具体方法和措施).
  • Audit (审计): Checks everything is being obeyed (检查各项规定是否被遵守).
  • Asset Registers (资产登记册): Determine the scope (决定了安全管理体系的适用范围).

[!IMPORTANT]
An Asset Register should include everything that can affect the security of information. If it’s not in the register, it’s outside the scope of the ISMS.
(资产登记册应包含所有可能影响信息安全的事物。任何未登记在册的资产，都不在 ISMS 的保护和管理范围之内。)

2. The Multi-Layered Asset Model (多层级资产模型)

Assets are not just datasets; they are vertical stacks of dependencies. (资产不仅仅是数据，它们是自上而下相互依赖的堆栈。)

Layer (层级) Example in Slides (课件示例) 中文解释
Data set (数据集) “The HR information” 核心业务数据（如人力资源信息）
Application (应用程序) “The Oracle instance” 处理数据的软件（如 Oracle 数据库实例）
Platform (平台) “Solaris Server number 6”, “EMC Array” 承载软件的操作系统或硬件平台
Locations (物理位置) “Birmingham data centre” 存放硬件的物理空间（如伯明翰数据中心）
Infrastructure (基础设施) “AD Server”, “UPS” 共享的网络或电力设施（如活动目录服务器、不间断电源）
People (人员) “HR Director”, “Cleaner” 涉及的各类人员（包括HR总监，甚至清洁工）

The Full Scope (完整范围的界定):

  • Everything you might want to record a threat against. (任何你可能需要记录其面临威胁的事物。)
  • Everything an attacker might want to control / destroy / compromise. (任何攻击者可能想要控制、破坏或妥协的事物。)
  • Everything that might stop working. (任何可能发生故障罢工的事物。)
  • Everything you might want to apply a control to. (任何你想要施加安全控制措施的事物。)

3. Asset Dependencies & Construction (资产依赖与构建方法)

3.1 Mapping Dependencies (映射依赖关系)

Systems are highly interconnected. For example, both an HR dataset and a Finance dataset might depend on the same Oracle instance, which runs on a Solaris server in the Data Centre. Meanwhile, a Web Server might run on Windows -> VMware -> Data Centre.
系统是高度互联的。所有这些最终都依赖于基础设施，如 AD (活动目录)、防火墙 (Firewall)、网络 (Networking) 和电力 (Power)。

3.2 Construction Approach: “Meet in the Middle” (构建路径：中间汇合法)

Building the register is an iterative process (构建登记册是一个迭代的过程).

  1. First Step (Top-Down / 自顶向下): Focus on systems and databases (Payroll, HR, source control). Then expand downwards to the hardware supporting them. (从业务系统如工资单、HR系统开始，向下梳理它们运行在什么硬件上。)
  2. Second Step (Bottom-Up / 自底向上): Focus on “what you can see”—the physical boxes (“tin”), buildings, people, and cabling. Then expand upwards to see what systems they host. (从看得见的物理设备、建筑、线缆开始，向上追溯它们承载了什么系统。)
  3. Result: Maybe meet in the middle. (最佳结果是双向并行，在中间层汇合，确保没有遗漏。)

3.3 Reality Intrudes & Iteration (现实的挑战与迭代)

  • Legacy Issues (遗留问题): You will find lots of things you didn’t know existed—legacy systems and legacy infrastructure. (你会发现很多你根本不知道存在的旧系统和老旧基础设施。)
  • Entropy (熵增/混乱度): “Mature” computer installations are subject to the 2nd law of thermodynamics (they naturally become more disordered over time). (“成熟”的计算机系统受热力学第二定律支配，即随着时间推移，系统会自然变得越来越混乱和无序。)
  • Simplification (精简): ISO 27001 is an opportunity to simplify. “Removal is better than controls.” (建立 ISO 27001 体系是一个精简系统的机会。直接移除无用的老旧资产，比为它们增加安全控制措施更有效。)
  • Iterate (持续迭代): You won’t get it right the first time. It’s OK to iterate and improve. (你不可能一次性做对，持续迭代和改进是正常的。)

4. 📝 Exercise: Sketching Dependencies (课堂练习：绘制依赖图)

Think about a business that has an HR system, a finance system, a public webserver and some engineering systems (source control). Sketch a graph like the one we’ve just looked at.
(思考一家拥有 HR 系统、财务系统、公共 Web 服务器以及一些工程系统（如源代码控制）的企业。试着画出它们之间的依赖关系图。)

Why is this exercise important? (为什么这很重要？)

  • Complex Mesh (复杂的网状结构): Real businesses have highly complex dependency meshes. Every year that goes by makes mapping it harder. (真实企业的依赖关系错综复杂，时间拖得越久越难梳理，所以必须尽早将其纳入安全管理体系。)
  • The Weakest Link (短板效应): New, secure systems locked in a data centre might rely on old, unsupported legacy hardware sitting under someone’s desk. If that legacy hardware fails, the whole system fails. (看似安全的现代系统，可能底层依赖着某人办公桌下一台无人维护的旧电脑。这就是系统的致命弱点。)

5. Security vs. Capital Asset Registers (安全资产清单 vs 财务固定资产清单)

Why isn’t this just the capital asset register, plus a bit? (为什么不能直接用财务部的固定资产清单稍微改改就当安全资产清单？)

  • Focus (关注点不同): Security registers start from systems and functions; Capital registers focus on licenses, bricks, and physical “tin”. (安全更关注系统功能，财务只关心买了几个授权、几台机器。)
  • Financial Thresholds (财务门槛): IT equipment is increasingly leased or very cheap (e.g., a £50 Raspberry Pi). Therefore, it is not capitalised but still poses a massive security risk. (很多 IT 设备是租来的，或者极其便宜，根本达不到财务记账的标准，但它们同样存在安全风险。)
  • Functionality (功能盲区): Financial registers usually don’t talk about what the equipment actually does. (财务清单不关心这台服务器是用来发工资的还是用来做测试的。)
  • Scrapping (报废滞后): Companies are bad at scrapping; financial registers often contain lots of equipment you don’t have or use anymore. (企业在设备报废销账上通常做得很差，财务清单里往往有一堆早就不用或丢了的“幽灵设备”。)

6. Cloud Assets & Shadow IT (云资产与影子 IT)

6.1 What is a Cloud Asset? (什么是云资产？)

If you are running a service on an AWS instance, what do you put into your ISO 27001 register?
(如果你在 AWS 上运行服务，你的资产登记册里该写什么？)

  • The asset is the Information and the Contract you have with Amazon. The audit is of the fitness for purpose of that contract.
  • (在云端，你失去了对物理硬件的控制。你的资产变成了“信息本身”以及“你与云服务商签订的合同”。审计的重点也变成了“审查这份合同是否满足了你的安全需求”。)

6.2 The “Risk Managed” Approach ("风险管理"的委婉说法)

Formal accreditations of cloud services are rare. Businesses using SaaS (like Salesforce) will adopt a “Risk managed” approach.
(云服务的正式审计认证很少见，企业使用 SaaS 服务时通常自称采用“风险管理”方法。)

  • Euphemism (委婉语): This is often a euphemism for relying on a contract, reputation, and good luck. (这通常是依赖“合同保障、大厂声誉和好运气”的委婉说法，因为小客户根本没有权限去审计 AWS 的数据中心。)
  • On-Prem Cloud (本地云): Because of this trust issue, large integrators and governments often prefer “On-Prem” cloud (bringing cloud-like services into their own physical data centers). (因为缺乏信任，大企业和政府更倾向于采用“本地云”，即在自己可控的机房里搭建类似云的服务。)

6.3 Maintaining the Register & Shadow IT (清单维护与影子 IT)

Once written, it’s NOT easy to maintain. (资产清单一旦写好，维护起来极其困难。)

  • New systems and new dependencies don’t require financial approval, or at least don’t appear as capital assets. (员工购买新的云服务订阅或便宜硬件通常不需要高层财务审批。)
  • This requires active co-operation from the business to avoid “Shadow IT” (员工私自使用的 IT 资源). (这需要业务部门的积极配合，否则防不胜防的“影子 IT”会带来巨大的未知风险。)
