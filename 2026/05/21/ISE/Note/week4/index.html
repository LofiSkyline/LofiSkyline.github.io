第四周：自动软件测试与基于搜索的软件测试 (AST & SBST)

本周课程从数据驱动的软件工程转向了优化问题（Optimization）。软件测试在本质上可以被看作是一个优化过程：我们的目标是优化质量指标，或者通过使软件系统表现“变差”（即触发失效）来寻找缺陷。

1. 软件测试的核心目标与覆盖率指标 (Coverage Metrics)

自动化测试生成的核心目标是寻找一个测试套件 (Test Suite)，其中的测试用例 (Test Cases) 能尽可能多地覆盖代码，同时保持运行速度快 (Fast to run)。理想情况下，测试套件应确保程序中的每个代码单元（路径、分支或语句）至少被覆盖一次。

1.1 分支覆盖 (Branch Coverage) vs. 路径覆盖 (Path Coverage)

  • 分支覆盖 (Branch Coverage)：

    • 定义：确保程序中每个控制流判定点（如 if 语句）的每一个可能结果（True 或 False）都至少被执行一次。
    • Case Study 1：奇偶数判断函数 checkEvenOdd
      • 测试用例 1：输入 4 $\rightarrow$ 覆盖 True 分支。
      • 测试用例 2：输入 7 $\rightarrow$ 覆盖 False 分支。
      • 仅需这 2 个用例即可达到 100% 全分支覆盖。

  • 路径覆盖 (Path Coverage)：

    • 定义：要求程序中所有可能的执行路径组合（Permutations）都被完全遍历一次。
    • Case Study 2：计算折扣的函数 calculate_discount
      • 分支覆盖要求：只需 2 个用例，如 (100, true, true) 和 (100, false, false) 即可把四个分支全走一遍。
      • 路径覆盖要求：必须遍历所有 4 个独立组合：(T, T), (F, F), (T, F), (F, T) 才能达到 100% 路径覆盖。

1.2 圈复杂度与分支统计 (Cyclomatic Complexity)

Case Study 3：BankAccount 类的总分支数量计算

  1. __init__：包含 0 个条件判定，0 分支。
  2. deposit(amount)：包含 if amount > 0 判定，2 分支（T/F）。
  3. withdraw(amount)：包含 if amount <= self.balance 判定，2 分支（T/F）。
    结论：类的总分支数是其所有函数分支数的累加，即 $0 + 2 + 2 = 4$ 个分支。

Case Study 4：嵌套分支的覆盖推导
假设有 $k=3$ 层深层嵌套。按照 $V(G) = k + 1$，只需 4 个用例即可全覆盖：

  • Case 1 (直通最底层)：(True, True, True) $\rightarrow$ 覆盖 Layer 1, 2, 3 的 True（“借道”刷卡）。
  • Case 2 (在最底层拐弯)：(True, True, False) $\rightarrow$ 覆盖 Layer 3 的 False。
  • Case 3 (在第二层拐弯)：(True, False, Any) $\rightarrow$ 覆盖 Layer 2 的 False。
  • Case 4 (在第一层拐弯)：(False, Any, Any) $\rightarrow$ 覆盖 Layer 1 的 False。
    结论：每一层只需额外增加一个对立用例去激活 False 走向，这种嵌套结构（最坏情况）仅需 $k+1$ 个用例。

2. 基于搜索的软件测试 (SBST) 与遗传算法 (GA)

2.1 术语映射：从生物学到测试工程

层级 生物学概念 测试领域的对应 实际例子 (Java)
1 等位基因 (Allele) 字面量 / 参数值 54 或 -20
2 基因 (Gene) 单条测试语句 int var0 = 54;
3 染色体 (Chromosome) 一个测试用例 (Test Case) 一个 @Test 方法
4 基因型 (Genotype) 测试套件 (Test Suite) 完整的测试文件 FooTest.java
5 表现型 (Phenotype) 动态行为结果 “85% 覆盖率，2 个 Crash”

2.2 演化流程 (Evolutionary Loop)

  1. 随机初始化：基于测试集群 (Test Cluster) 静态扫描生成符合强类型约束的随机代码（例如随机生成 obj.deposit(amount) 前会强制插入实例化语句）。
  2. 评估：通过代码插桩记录分支触发情况，利用适应度函数打分。
  3. 锦标赛选择 (Tournament Selection)：随机抓取 $k$ 个文件单挑，高分者晋级，保留随机性以防止早熟收敛 (Premature Convergence)。
  4. 代码切片重组 (Crossover)：交换家长间的部分测试函数（测试用例）。
  5. AST 级代码变异 (Mutation)：抛掷骰子，等概率（各 1/3）互斥执行删除 (Remove)、修改 (Change) 或插入 (Insert)。
  6. 淘汰与迭代：按得分全局排序，执行精英保留和末位淘汰，维持种群规模恒定。

3. 适应度函数：从单目标到多目标

3.1 单目标优化 (EvoSuite 逻辑)

在 EvoSuite 中，适应度函数是惩罚函数，目标是使分数趋近于 0：
$$fitness(T) = \underbrace{|M| - |M_T|}{\text{未覆盖分支数}} + \underbrace{\sum d(b_k, T)}{\text{分支距离 (Branch Distance)}}$$
如果两个文件覆盖率相同，算法优先选择代码更精简 (less statements) 的文件。

3.2 多目标优化 (Sapienz 逻辑)

在 Sapienz 中，$a$ 和 $b$ 的比对指标升级为最高层优化目标向量 (Objective Vector)：
$$a = [a_1, a_2, a_3] = [\text{覆盖率}, \text{总步数}, \text{崩溃数}]$$

目标的归一化与统一 (Uniformity)：
算法将所有目标统一改写为越小越好的 Loss/Penalty 值：

  • 覆盖率：改写为 $\text{未覆盖比例}$。趋近于 $0.0$ 代表全覆盖。
  • 总步数：直接使用总点击数。趋近于较小值代表精简。
  • 崩溃数：改写为 $\frac{1}{1 + \text{崩溃次数}}$。趋近于 $0.0$ 代表抓 Bug 极多。

3.3 帕雷托支配 (Pareto Dominance)

文件 $a$ 想要强支配 (Dominate) 文件 $b$，必须同时满足：

  1. 不比你差：在所有 3 个指标上，$a_j \le b_j$。
  2. 至少有一项碾压：至少存在一个 $j$，使得 $a_j < b_j$。
    弱支配： 完全相等的两个解 互相弱支配对方
    等级分类 (Fronts)：
  • Level 1 ($F_1$ 前沿面)：全缓冲区内没有被任何人支配过的精英文件。
  • Level 2 ($F_2$ 前沿面)：仅被 $F_1$ 支配的文件。
  • 淘汰时优先保留 $F_1$，并利用拥挤距离 (Crowding Distance) 确保测试用例的多样性。

4. Android 自动化测试：Sapienz 专项技术

Android 是事件驱动 (Event-driven) 系统，存在状态残留。

4.1 原子事件 vs. 基调事件

  • 原子事件 (Atomic Event)：单一原生指令（Touch, Motion, Rotation）这是最底层的、单一的系统原生操作指令。包括：Touch（点击特定坐标）、Motion（手势滑动）、Rotation（物理屏幕旋转）、AppSwitch（切换前后台应用）等。
  • 基调事件 (Motif Event)：这是 Sapienz 能够跑出商业级高覆盖率的“作弊大招”。它是一个自动封装的复合操作块（Compound of several inputs） ：
    • 它会分析当前的手机屏幕，自动找到所有的文本输入框（Text Fields），并在里面自动填充合法的字符串
    • 紧接着，它会自动寻找当前屏幕上所有可以点击的组件（Clickable widgets），强行尝试触发它们，以此来帮应用安全地转移跳转到下一个新屏幕（Next view）

4.2 Sapienz 的三大优化目标 (Objectives)

  • 🎯 目标一：最大化代码覆盖率 (Maximise Coverage)
    Sapienz 提供了三种粗细不同的运行时插桩记录方式 ：

    • Statement Coverage（语句覆盖率）：细粒度，统计执行到了多少行具体的 Android 底层渲染或业务代码。
    • Method Coverage（方法覆盖率）：中等粒度，统计调用了多少个业务函数。
    • Android Activity Coverage（Activity 覆盖率）：粗粒度，最符合人类直觉。Activity 是 Android 系统的逻辑屏幕单元。例如一个拨号 App 包含四个核心 Activity 屏幕：Dialer（拨号键盘）、Contacts（联系人列表）、View Contact（查看联系人）、New Contact（新建联系人）。算法通过统计测试序列成功解锁并点进去了多少个独立屏幕，来评估覆盖率分数。

  • 🎯 目标二：最小化测试序列总长度 (Minimise Length)
    算法要求测试步骤越精简越好，步数的累加公式为：
    $$\text{Total Length} = \sum (\text{各事件的步数强度})$$

    • 如果在序列里执行了一个原子事件（比如晃一下屏幕），长度加 1。
    • 如果执行了一个基调事件，长度加 = [输入的字符串个数 + 触发的点击按钮数]。

  • 🎯 目标三：最大化捕获崩溃数量 (Maximise Crashes)
    系统会在 Android 后台挂载一个异常捕获钩子，一旦 App 发生闪退、黑屏、ANR（应用无响应）或者抛出 NullPointerException 等未捕获异常，就判定为抓到了一个 Crash（崩溃）缺陷。抓到崩溃的事件序列文件会获得极高的豁免权（在这一维度具备极低的损失分，从而极难被淘汰）。

4.3 高层级变异 vs. 低层级变异

  • High-level Mutation (高层级变异)：
    • 操作粒度：函数级别 (Function level)。
    • 动作：随机打乱不同测试用例（测试函数）之间的执行先后顺序 (Shuffle)。
    • 目的：探测由于 App 状态残留 (State persistence) 引发的深层崩溃。
  • Low-level Mutation (低层级变异)：
    • 操作粒度：语句/指令级别 (Statement level)。
    • 动作：深入函数内部执行同长替换、异长插入或序列内洗牌 (Shuffling)。
    • 目的：通过毫秒级的乱序操作，探测并发冲突 (Concurrency) 引发的闪退。
