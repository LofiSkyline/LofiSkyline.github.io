1. 初始阶段：从未确认到确认

  • UNCONFIRMED（未确认）

    • 背景： 当 end-users（终端用户）或测试人员刚在 JIRA 或 GitHub 等 Tracker 上提交一个新的缺陷时，系统默认给出的初始状态 。

    • 流转： 此时，该报告还没有经过官方的验证。如果分拣人员（Triager）或开发人员复现了这个问题，确认这确实是一个 Bug（Indeed a bug），它就会被推进到下一个状态 。

  • NEW（新提交）

    • 背景： 报告已被确认是软件本身的缺陷 。

    • 流转： 它会进入待办列表（Backlog），等待被分发。如果在这个阶段发现报告有误，或者负责人发生了变更，它的所有权可能会被修改（Ownership is changed） 。

2. 处理阶段：指派与修复

  • ASSIGNED（已指派）

    • 背景： 某位特定的开发人员接管了（takes possession of）这个 Bug 。他的任务就是去阅读 Description、分析 Code、并在本地定位和修复它 。

    • 流转： 当开发人员编写完补丁（Patch）并认为缺陷已经解决（The bug has been solved / Developer has solved the bug）后，他会将状态修改为 RESOLVED 。

3. 解决阶段：五种不同的 Resolution

当 Bug 报告进入 RESOLVED（已解决） 状态时，必须指定一个具体的 Resolution（解决结论） 。课件中列出了 5 种最经典的结论 ：

Resolution 结论 含义 (Meaning)
FIXED 真正修复。代码已被修改，补丁已提交 。
WONTFIX 不予修复。可能因为该行为是设计使然（It’s a feature），或者修复成本过高、影响范围太大。
WORKSFORME 无法复现。开发人员在自己的环境下测试发现一切正常，无法重现报告中描述的错误。
DUPLICATE 重复报告。这个 Bug 已经被别人先一步提交了（这正是第 51 页“重复报告检测”要解决的问题） 。
INVALID 无效报告。用户描述错误、使用了错误的配置，或者根本不是软件本身的问题。

4. 验证与关闭阶段

  • VERIFIED（已验证）

    • 背景： 缺陷被标记为 RESOLVED 后，不能直接关闭，必须由测试人员（Testers）或项目负责人在特定的构建（Build）中进行回归测试。

    • 流转： 如果测试表明解决方案确实有效（Solution worked），Bug 状态就会推进到 VERIFIED 。如果在后续的观察中没有再次收到该缺陷的报告（No occurrence of the bug is reported），它就会被推进到终态——CLOSED（关闭） 。

5. 循环流转：重新打开 (REOPEN)

在整个生命周期中，还有一个非常重要的“逆向流转”状态—— REOPEN（重新打开） ：

  • 触发条件 1： 当 Bug 被标记为 RESOLVED，但测试人员测试后发现解决方案并不令人满意（Solution is not satisfied），即 Bug 依然存在 。

  • 触发条件 2： 当 Bug 已经走到 VERIFIED 甚至 CLOSED 状态，但在新版本中该 Bug 再次复现（Bug is reopened） 。

  • 流转： 一旦被 REOPEN，报告就会重新回到 ASSIGNED 或 NEW 状态，由开发人员重新介入排查 。

为了应对考试中 “简要描述其工作原理（Briefly understand how they work）” 的要求，你需要掌握这三个经典模型最核心的工作机制（Mechanism）和独特性质（Characteristics）。

在答题时，用英文专业词汇（Keywords）精准命中得分点至关重要。以下是为你整理的冲刺版背诵框架：

1. Decision Tree (DT - 决策树)

  • 工作方式 (How it works): 它是一种树状结构 (Tree structure) 。模型通过在每个节点上对特征属性进行测试/切分 (Split/Test) ，根据不同的特征值将数据分流到不同的分支 (Branch) ，层层向下流转，直到最终的叶子节点（Leaf node）输出分类标签（如 Bug 或 非 Bug） 。

  • 特性 (Characteristics):

    • 直观易懂 (Highly interpretable): 它的运行逻辑非常符合人类的条件判断思考方式（If-Then 逻辑） 。

    • 非线性 (Non-linear structure): 能够天然地处理复杂的非线性关系。

2. Naive Bayes (NB - 朴素贝叶斯)

  • 工作方式 (How it works): 它是基于贝叶斯定理 (Bayes theorem) 的概率分类器 。模型会计算在给定的文本特征 $x$ 下，该报告属于每一个类别 $C_k$ 的条件概率 (Conditional probability) 。计算完所有类别的概率后，选择概率最高 (Higher probability) 的那一类作为预测结果 。

  • 特性 (Characteristics):

    • “朴素”假设 (Naive assumption): 它有一个强假设，即假设所有的文本特征（即词汇之间）是相互独立 (Feature independence) 的。虽然这个假设在现实中往往不成立，但该模型在文本分类任务中却异常高效。

    • 计算快 (Computationally efficient): 只需要进行简单的计数和概率乘法，训练和预测速度极快。

3. Logistic Regression (LR - 逻辑回归)

  • 工作方式 (How it works): 它是一个用于分类的线性模型 (Linear model) 。它首先将输入的自变量 $x$ 进行线性组合 ，然后通过一个 Sigmoid 激活函数 (Sigmoid / Logistic function) 将结果映射为一个 $[0, 1]$ 之间的概率值 $p(x)$ 。最后利用一个截断点/阈值 (Cutoff point) 来决定最终归属于哪一类 。

  • 特性 (Characteristics):

    • 输出概率值 (Outputs probabilities): 它不仅给出分类，还能给出模型对这个分类的信心程度（即概率值） 。

    • 易于优化 (Easy to train): 它是统计学中的经典方法，对线性可分的数据集表现极佳。

💡 考试答题套路提示

如果考题要求对比它们，记住一句话：

DT 靠“层层规则切分” ，NB 靠“算条件概率” ，而 LR 靠“线性映射加 Sigmoid 函数” 。它们在 Bug 分类中，都是为了处理由文本词频（TF）构成的特征向量 。
