SSMLecture 7: Device Security - Trusted Boot and Code Signing

SSMLecture 7: Device Security - Trusted Boot and Code Signing

1. Introduction & The Threat Model

Device security focuses on the security of the code running on the device, ensuring it hasn’t been tampered with or replaced.

1.1 The Rootkit Challenge (启动前篡改)

A Rootkit is a particularly dangerous type of malware designed to gain root privileges and, crucially, it often loads pre-system load (before the Kernel).

  • Stealth: Since it controls the foundation, it can hide its existence from the OS and antivirus.
  • Privilege: It can disable security protections and install backdoors.
  • Logic: If a malicious actor replaces the firmware or bootloader, the software layers above cannot detect the breach because their “ground truth” is already compromised.

1.2 Defense in Depth (深度防御)

To counter these threats, we apply multiple layers:

  1. Securing the Boot Process: Establishing a Chain of Trust from hardware up.
  2. Securing the User Space: Sandboxing and runtime protections.
  3. Securing Distribution: Code signing and trusted app stores.

2. Cryptographic Foundations: Efficiency vs. Security

The entire device security model relies on the balance between Hash functions and Digital Signatures.

2.1 The Engineering Trade-off

  • Hash Functions (Integrity): Extremely fast and efficient. Used to “fingerprint” large data blobs (M). Ideal for checking if a 10GB partition has changed.
  • Digital Signatures (Authentication): Computationally expensive (slow). Used to prove “who” produced the data. Due to its cost, signatures are typically only applied to Head Info or Meta Info (like a Root Hash).

2.2 The Combined Mechanism (签署与验证闭环)

  1. Signing: Data $\rightarrow$ Hash $\rightarrow$ Encrypt Hash with Private Key $\rightarrow$ Signature.
  2. Verifying:
    • Device re-computes the Hash of the received data.
    • Device decrypts the Signature using the Public Key to get the “Official Hash”.
    • If they match, the data is both Authentic (from the owner) and Intact (not modified).

3. Securing the Boot Process: Chain of Trust

The boot process is a “relay race” where each stage verifies the next before execution.

3.1 Root of Trust (RoT, 信任根)

The RoT is the foundation that we trust unconditionally. It consists of:

  • Storage: Secure location for keys.
  • Keys: Public/Private keys.
  • Algorithms: Built-in Hash and Signature functions.
  • Certificates: Proving the key belongs to the OEM (e.g., Apple/Google).

Hardware RoT (laid down during chip fabrication) is the most secure because it is immutable (不可更改), unlike software-based RoT stored in NAND flash.

3.2 Trusted Boot Flow

  • Step 1: Hardware RoT verifies the Bootloader.
  • Step 2: Bootloader initializes the system and verifies the Kernel.
  • Step 3: Kernel verifies the System Partition.
  • Fail Secure: If any stage fails verification, the process stops to prevent the system from running in a compromised state.

4. Efficient Verification: dm-verity & Merkle Trees

Verifying a multi-gigabyte system partition at every boot is impractical for battery life and speed.

4.1 Merkle Trees (哈希树)

Instead of signing a whole giant file, we use a tree structure:

  • Layer 0: Data is split into 4KB blocks, each hashed individually.
  • Intermediate Layers: Hashes are paired and hashed again, forming a tree.
  • Root Hash: The single hash at the top representing the entire partition.
  • Optimization: Only the Root Hash needs a digital signature. (只需签名根哈希，即可锁定万千碎块的安全性)

4.2 dm-verity

This is the kernel implementation that uses the Merkle Tree for On-demand Verification:

  • It only verifies a block when it is actually read into memory.
  • If a Rootkit modifies a single bit in a 4KB block, its hash will mismatch the Merkle Tree path, and dm-verity will block the read.

5. Hardware-Anchored Security

Moving the RoT into specialized hardware makes it a “Black Box” that external software (even a compromised CPU) cannot peek into.

5.1 TPM (Trusted Platform Module)

A discrete, independent chip.

  • PCR (Platform Configuration Registers): The “memory” of the boot process.
    • Extend-Only: PCRs cannot be written to directly; they can only be “extended” ($New = Hash(Old + Data)$).
    • Implication: A黑客 cannot hide a BIOS change by “resetting” the log; any change permanently alters the final PCR value in an unpredictable way.
  • Sealing: TPM can “seal” keys so they are only released if the current PCR values match the “known good” boot state.
    TPM 启动方式

5.2 TEE (Trusted Execution Environment) - ARM TrustZone

  • Logical Isolation: Partitions the CPU into a Secure World (SW) and a Normal World (NW).
  • Risk: Since they share physical resources (Cache, Pipeline), TEE is potentially vulnerable to Side-channel attacks that guess register states or keys.
    TEE条件启动方式

5.3 Apple Secure Enclave Processor (SEP)

  • Physical Isolation: Unlike TEE, SEP has its own dedicated RAM, OS (SEPOS), and Hardware Filter.
  • Secure Mailbox: Communication with the Application Processor (AP) happens via an encrypted mailbox. Even if the iOS kernel is fully rooted, the attacker cannot “dump” the keys inside the SEP.
    SEP启动方式

6. Application Security

Once the OS is up, multiple software-level defenses are enforced:

  • Code Signing: Only authorized apps (signed by a store certificate) can run.
  • Sandboxing: Restricts apps to their own home directories (using unique UIDs on Android).
  • Mandatory Access Control (MAC): SELinux/AppArmor defaults to “deny” for all API calls unless explicitly permitted.
  • Memory Protection:
    • ASLR: Randomizes memory layout to prevent reliable exploits.
    • XN (Execute Never): Marks data areas (stack/heap) as non-executable to prevent code injection.

Summary Table: Comparison of Hardware Security

Mechanism Isolation Type Resource Sharing Main Strength
TPM Physical (Discrete Chip) None Immutable logs (PCR), Sealing.
TEE Logical (CPU Mode) Shared Cache/Pipeline Fast, low cost, efficient.
SEP Physical (Dedicated Core) None (Hardware Filter) Hybrid of speed and physical isolation.
