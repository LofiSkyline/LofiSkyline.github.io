DDS-06: Consensus (共识问题)

Consensus 是分布式计算中最核心、最著名的问题之一。其本质是一个 Decision Problem（决策问题）：系统中一组分布式的进程需要就某个值达成共同的决定，即使它们最初的提案（Initial Opinions）各不相同。

1. 共识问题的形式化定义 (Formal Definition)

共识问题要求所有 Correct Processes（正确运行的进程）最终决定同一个值。

1.1 基本属性 (Core Properties)

无论在哪种故障模型下，共识算法都必须满足以下三个基本条件：

  1. Agreement (达成一致): No two processes decide on different values. (任意两个进程不得决定不同的值。)
  2. Validity (有效性): 防止产生“平凡解”（Trivial Solutions，如算法永远固定输出 0）。
    • 基本要求: 如果所有进程初始值都为 $v$，则唯一可能的决定值必须是 $v$。
    • 在不同故障模型下，其强弱程度会有所调整。
  3. Termination (终止性): All processes eventually decide. (所有进程最终必须做出决定。)

2. 故障模型定义 (Fault Models)

在研究共识时，系统的故障模型（Fault Model）决定了算法的定义和可解性。

2.1 Link Failure (链路故障 / Coordinated Attack Problem)

  • Definition: 消息在传输过程中可能会丢失（Messages can be lost），但进程本身不会出错。
  • 典型案例: Coordinated Attack Problem (协调攻击问题)。
    • 两支军队必须同时进攻才能胜利。如果一支进攻而另一支撤退，则会失败。
    • 通讯兵（Messenger）穿越敌占区可能被捕（消息丢失）。
  • Impossibility Result: 在存在 Link Failure 的同步网络中，不存在能够 100% 正确达成共识的算法。
    • 原因直觉: 无论确认（Acknowledgment）多少次，发送最后一条消息的进程永远无法确定对方是否收到了该消息。这种不确定性通过“不可区分执行（Indistinguishable Executions）”可以证明会导致违反 Agreement 或 Validity。

2.2 Stop Failure (停止故障 / Crash Failure)

  • Definition: 进程在运行过程中可能随时停止工作（Stop executing and sending messages forever），一旦停止，它就不会再恢复。
  • 假设: 链路是可靠的（Reliable Links），即只要发送者没崩溃，消息一定能送达。
  • Consensus in Stop Failure Context:
    • Agreement: 任意两个进程（包括后来崩溃的）不能决定不同的值。
    • Validity: 如果所有进程初始值都为 $v$，则 $v$ 是唯一可能的决定值。
    • Termination: 所有非故障（Non-faulty/Active）进程最终决定。

3. Stop Failure 下的共识算法：FloodSet

FloodSet 是解决 Stop Failure 模型下共识问题的经典算法族。其核心思想是“信息洪泛”，让每个进程尽可能掌握系统中存在的所有初始值。

3.1 算法流程 (Algorithm Steps)

  1. Initialization: 每个进程 $i$ 维护一个集合 $W_i$，初始值为其自身的提案 ${v_i}$。
  2. Flooding (执行 $f+1$ 轮):
    • 在每一轮中，每个存活的进程将其 $W_i$ 广播给所有进程。
    • 进程接收来自他人的集合，并将其并入（Union）自己的 $W_i$。
  3. Decision Rule (决策规则):
    • 在第 $f+1$ 轮结束后，进程 $i$ 检查 $W_i$。
    • 规则: 如果 $W_i$ 是单元素集合（Singleton Set, $|W_i|=1$），则决定该唯一值；否则，决定一个预设的默认值 $v_0$。

3.2 为什么是 $f+1$ 轮？(Core Reasoning)

这是该算法最关键的逻辑考点。如果在系统中最多容忍 $f$ 个进程发生 Stop Failure，那么必须执行 $f+1$ 轮。

  • Pigeonhole Principle (抽屉原理): 在 $f+1$ 轮通信中，如果总共只有 $f$ 次失效，那么至少存在一轮（Clean Round）没有任何进程发生新的失效。
  • 推理过程:
    • 假设第 $r^*$ 轮是 Clean Round。
    • 在这一轮开始时，所有仍存活的进程集合为 $P_{active}$。
    • 由于这一轮没有进程崩溃，所有 $P_{active}$ 中的进程都会成功收到彼此的消息并完成 $W$ 的并集更新。
    • 结论: 在第 $r^*$ 轮结束时，所有存活进程的 $W$ 集合将变得完全相同。
    • 在随后的轮次中（从 $r^*+1$ 到 $f+1$），由于 $W$ 已经一致，且没有新值进入系统，所有存活进程将保持这个一致的 $W$ 直到最后。
  • Case Analysis:
    • 如果只执行 $f$ 轮，可能每一轮都有一个进程在发送消息的中途崩溃（只发给了部分人），导致最后活下来的进程手中持有的 $W$ 互不相同，从而违反 Agreement。

3.3 复杂度分析 (Complexity Analysis)

假设系统有 $n$ 个进程，最多 $f$ 个故障，初始值用 $b$ bits 表示：

  • Round Complexity: $f+1$ 轮。
  • Total Messages: $O(n^2(f+1))$。每一轮每个进程发给 $n$ 个人。
  • Message Size: 在原始 FloodSet 中，集合 $W$ 最多包含 $n$ 个不同的初始值，因此单条消息大小为 $O(nb)$ bits。
  • Communication Complexity (Total bits): $O(n^3 b (f+1))$。

4. 优化：Reduced FloodSet (降低通信量)

通过对决策规则的洞察，我们可以大幅降低通信复杂度。

4.1 核心洞察 (Insight)

在决策阶段，进程其实并不关心 $W$ 集合里具体有哪些值，它只关心：

  1. 是否有且仅有一个值？
  2. 还是存在多个值？（只要存在超过 1 个值，就会 Fallback 到默认值 $v_0$）。

因此，每个进程一生中只需要广播两次非空消息：

  1. First Broadcast: 第一轮广播自己的初始值。
  2. Second Broadcast: 在后续某轮中，一旦第一次发现了一个与自己初始值不同的值，就立即广播这个“冲突信号”。此后再收到任何值都保持沉默。

4.2 优化后的复杂度

  • Rounds: 依然是 $f+1$ 轮。
  • Total Messages: $O(n^2)$。每个进程最多向其他进程发送 2 条消息。
  • Message Size: $O(b)$ bits（只发送单值而非集合）。
  • Total Communication Complexity: $O(n^2 b)$。成功消除了与 $f$ 的关联。

5. EIG (Exponential Information Gathering)

EIG 是一种结构化的方法，通过构建 EIG Tree 来记录信息的传播路径（即“谁告诉谁某个值”）。

  • Pros: 能够比 FloodSet 提供更丰富的信息结构，对于处理更复杂的故障（如 Byzantine）非常重要。
  • Cons: 结构复杂，树的节点数随轮次指数级增长，在 Stop Failure 下显得“性能过剩”。
  • Decision Rule: 同样在 $f+1$ 轮后，根据树中的信息应用规则（如取最小值或最大值）达成共识。

6. 总结与对比 (Pros and Cons)

特性 Link Failure (Coordinated Attack) Stop Failure (FloodSet)
可解性 Impossible Possible
核心挑战 消息丢失带来的无穷递归确认 进程崩溃导致的信息不对称
解决方案 放宽定义（概率性共识） 执行 $f+1$ 轮确保至少一个 Clean Round
主要优点 简单、易于理解其局限性 在同步网络下非常健壮且高效
主要缺点 无法保证 100% 安全 轮次开销大，依赖同步网络假设

  • 牢记 Consensus 的三个定义（Agreement, Validity, Termination）。
  • 深刻理解 $f+1$ 轮的原因及其背后的 Pigeonhole Principle。
  • 区分原始 FloodSet 与优化后 FloodSet 在消息数量上的量级差异。
